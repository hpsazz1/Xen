# P0-6 影子灰度入口与安全回退

## 1. 背景

P0-5 自适应 Kalman 候选冻结为 `688/810`，仍为 `HOLD_SHADOW`。按 `docs/068基础移动与预测重构复核计划20260714.md`，P0-6 只能先进入 DML/NDI 影子采集，不能启用正式 active。

就绪审计确认：运行时已经支持 `legacy/shadow/active` 请求模式，`active` 会安全降级为 `shadow`；影子帧具备时间分解、NIS、协方差、反馈/前馈/提前/积分、轨迹输出、命令抑制和构建身份字段；单元测试也证明影子链不会生成可排队的设备命令。但 UI 没有模式入口，非法配置值也会原样保存，不符合现场一键切换和安全回退要求。

## 2. 设计

本轮只补齐灰度入口，不改变控制器或设备语义：

1. Movement 页面新增“新链路灰度模式”。
2. 可选择 `legacy`、`shadow`、`active` 请求状态。
3. UI 明确显示 P0-5 仍为 `HOLD_SHADOW`；`active` 只是请求态，运行时有效模式仍为 `shadow`。
4. `legacy` 继续运行 r30 正式旧链路，新估计与控制链不运行。
5. 配置加载只接受三个合法值；未知值强制回退 `legacy`，避免拼写错误静默进入新链路。

## 3. 数据流与安全边界

```text
UI/配置请求模式
  ├─ legacy → effective=legacy → r30正式输出
  ├─ shadow → effective=shadow → 同帧CSV诊断，commandSuppressed=1
  └─ active → effective=shadow → 同帧CSV诊断，commandSuppressed=1
```

`AimPipelineRuntime` 仍不持有设备队列接口；轨迹输出无论透传或梯形模式都再次写入 `commandSuppressed=true`。本轮没有递增控制器修订号，保持 r30。

## 4. 测试

- 配置测试新增非法 `aim_pipeline_mode=typo` 回退 `legacy`。
- UI 可见性测试新增 `##aim_pipeline_mode` 灰度控件检查。
- 既有算法测试继续覆盖：
  - shadow 处理诊断但不产生设备命令；
  - active 请求降级为 shadow；
  - CSV 同时记录请求模式、有效模式、active 可用性和命令抑制；
  - 切回 legacy 与目标丢失/切换会重置影子状态。
- DML Release 正式脚本构建成功，DML `ctest` 8/8 通过。

## 5. P0-6 第一步实机约束

1. 只选择 `shadow`，不要选择 `active` 作为正式控制依据。
2. 轨迹模式保持 `off`，速度前馈、积分和经验提前保持 0。
3. 使用 DML/NDI 依次采集 static、left、right、reverse、jump。
4. 每份 CSV 必须确认：`RequestedMode=shadow`、`EffectiveMode=shadow`、`ActiveAvailable=0`、`CommandSuppressed=1`，且构建身份一致、无 `-dirty`。
5. 首轮只比较时间分解、NIS、协方差和候选请求，不根据单轮主观观感修改参数。

## 6. 后续

只有 DML/NDI shadow 连续三轮方向一致且 P0-5 硬门槛通过后，才能设计 P 反馈 active。届时必须新增设备发送失败自动回退，并递增控制器修订号；本提交不包含这些行为。

# P0-2角度坐标与延迟相机响应

## 背景与目标

P0-1 已把捕获、推理、控制和设备发送时间统一到同一 `steady_clock` 时间域，但旧运动补偿仍把设备 counts 视为发送后立即出现在画面。真实链路包含输入处理、游戏渲染、采集和推理延迟；若在观测时刻提前扣除尚未反映到画面的相机旋转，会把静止目标误判为运动目标。

本阶段在不改变 r30 正式控制语义的前提下，为新链路建立二维视线角坐标和固定纯延迟相机响应模型。输出仅写入 shadow 诊断，不进入设备命令计算，也不进入 P0-3 的 Kalman 状态估计。

## 排查结果

- 检测 pivot 位于完整源画面的中心坐标系，但旧补偿历史保存的是按当前画面换算的像素增量，无法跨 FOV、源分辨率和裁剪条件稳定比较。
- 旧 `ViewMotionHistory` 在设备成功发送时立即累计像素补偿，没有使用 P0-1 已贯通的设备发送时间与观测时间关系。
- r44 已基于六轮 DML/NDI 数据把正式 `ViewMotionHistory` 接入独立的 12 ms 命令到画面延迟；shadow 角度模型仍保留独立 60 ms 候选，详见 `docs/105同方向指令脉冲与补偿延迟修复20260716.md`。
- 被动 Profile 标定可输出延迟候选，但其有效性依赖现场激励和置信度；P0 阶段不能把一次标定结果自动写入正式控制配置。
- X/Y 设备比例可能不同，且 FOV 缩放可能随游戏 Profile 变化，因此每条成功命令必须保存发送时的独立 `degreesPerCountX/Y`，不能在查询历史时用当前配置重算。

## 坐标与符号约定

`AimCoordinateSpace` 新增完整源画面的二维透视换算：

```text
yaw = atan(2 * pixelOffsetX / sourceWidth * tan(HFOV / 2))
pitchDown = atan(2 * pixelOffsetY / sourceHeight * tan(VFOV / 2))
```

- yaw 向右为正。
- pitch 使用 `pitchDown`，向下为正，与屏幕 Y 和设备 Y counts 方向一致。
- `sourceWidth/sourceHeight` 必须来自完整源尺寸；缺失时才回退到已知屏幕尺寸，禁止把 320 检测裁剪宽度当作完整 FOV 跨度。
- 反向换算使用同一透视模型，单元测试验证二维像素与角度往返。

## 固定延迟响应模型

`AppliedViewMotionModel` 只接收底层设备确认成功后的命令。对发送时间为 `sendTime` 的命令：

```text
deltaYaw = countsX * degreesPerCountX
deltaPitch = countsY * degreesPerCountY
effectiveTime = sendTime + commandToFrameDelay
```

在 `effectiveTime` 之前，该命令不计入已出现在画面的相机角；到达该时刻后完整计入。P0-2 明确使用 `responseTau=0` 的纯延迟阶跃，不提前实现一阶响应、Smith 预测或自适应延迟。

历史同时按两秒窗口和 512 条上限裁剪。每条样本保存累计角，裁剪旧样本后仍保留首条剩余样本之前的累计基准，避免查询结果因历史裁剪发生跳变。配置变化会清空模型，防止不同延迟假设下的历史混用。

## Shadow 数据流

设备成功发送路径同时保留旧即时像素补偿和新延迟角度历史，因此 r30 行为不变。shadow 帧按以下时间关系计算：

```text
measuredLos = rawPivot相对画面中心的yaw/pitch
stabilizedLos = measuredLos + cameraAngle(observationTime)
relativeError = stabilizedLos - cameraAngle(controlTime)
```

`stabilizedLos` 表示消除程序已施加相机旋转后的稳定视线角；`relativeError` 再扣除观测到控制期间已经生效的相机角变化，为 P0-3 的时间对齐相对视线状态提供输入语义。每帧观测开始时会清空上一帧 `viewMotion`，避免无效帧沿用旧诊断。

## 配置与诊断

新增根配置 `aim_shadow_command_to_frame_delay_ms`，默认 `60 ms`，读取范围限制为 `0–250 ms`。该值只用于 shadow 模型，不自动采用被动 Profile 标定结果，也不覆盖 `[Games]`。

流水线 CSV 新增：

- `ViewMotionShadowValid`、`CommandToFrameDelayMs`；
- `DegreesPerCountX/Y`；
- `MeasuredLosYawDegrees`、`MeasuredLosPitchDownDegrees`；
- 观测时刻与控制时刻的 `AppliedCamera*Degrees`；
- `StabilizedLos*Degrees` 与 `RelativeError*Degrees`。

这些字段与 r30 的 `FinalMx/FinalMy` 写在同一帧，便于离线比较，但不改变命令队列、限速、预测或旧即时运动补偿。

## 验证结果

- 角度坐标测试覆盖二维透视往返和正负 pitch 符号。
- 响应模型测试覆盖延迟前后阶跃、正负 counts、X/Y 独立比例、历史上限和裁剪后累计基准。
- 合成 static 场景使用真实 `60 ms` 相机延迟；正确模型的稳定角 RMSE 和最大角速度为零，故意使用 `0 ms` 错误延迟时 RMSE 明显升高。
- 配置测试覆盖默认值、保存输出和 `250 ms` 上限。
- CSV 测试确认新增字段存在且与帧诊断一起输出。
- 修改提交前 DML Release 与 CUDA Release 均构建通过，CTest 均为 `8/8`；提交后将按构建身份规则再次双后端复建并核对。

## 边界与后续建议

本阶段不修改控制器修订号，仍为 r30；`active` 请求仍安全降级为 shadow。固定 `60 ms` 只是可审计初值，不代表所有设备、游戏、帧率和采集后端的最终延迟。

下一阶段 P0-3 应直接消费本阶段定义的角度与时间语义，建立 `[losAngle, relativeLosRate]` Kalman 影子链，并输出 innovation、NIS 和协方差。只有实机与回放数据证明固定延迟不足时，才评估一阶响应或更复杂模型，不能在 P0-3 前提前叠加。

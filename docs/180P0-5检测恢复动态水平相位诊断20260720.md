# P0-5 检测恢复动态水平相位诊断

## 背景

`docs/179P0-5已提交端点守卫重评20260720.md`在统一保留迟到命令的物理队列契约上，将四端点共同通过域从477提升到485且无原通过域损失，但收益全部来自static；left/right共同通过仍只有5/2个域。上一轮因此要求继续诊断检测恢复后的首批请求，区分估计器重置、已提交端点残差和限速相位，禁止继续围绕端点守卫堆叠局部补丁。

本轮直接复用14/0、14/2、15/0、15/2端点守卫正式输出的`cross_domain_frames.csv`。不修改运行时控制器、配置、UI、active能力、r64修订号或1440 counts/s二维设备上限。

## 分析设计

新增`tools/analyze_detection_resume_phase.ps1`，只读取逐帧跨域CSV：

1. 仅分析left/right，按`Scenario + Variant`分组，并使用有效帧间隔中位数推断该组正常控制周期。
2. 相邻有效帧间隔超过正常周期的1.5倍时，记为一次检测恢复；不依赖固定FPS，也不把文件首帧误判为恢复。
3. 默认导出恢复后的12帧，逐帧保留估计角度/速率、估计真值偏差、物理终点残差、限速前后比例、守卫前请求、守卫是否生效及实际请求。
4. 按物理端点和方向汇总恢复事件数、首帧零速率率、请求方向正确率、首帧残差、首帧限速比、连续饱和帧数、估计速率恢复帧数和守卫生效占比。

分析器只形成机器可读证据，不把诊断阈值接入`EvaluateGate`，也不自动提出控制候选。合成回归覆盖双端点、双方向、时间空洞、零速率冷启动、连续限速、方向判定、守卫生效和CSV输出。

## 结论与下一步

1. 控制器重置不是当前left/right动态阻断的主因。约4.85秒检测空洞后清除旧速率是必要安全行为；首帧0速率没有造成错向，且估计在2～3帧恢复。跨长空洞保留旧估计会引入不可验证的陈旧状态，不形成候选。
2. 恢复窗口的主导相位是大端点残差下的物理限速：首帧请求约为单帧预算的2.47倍，四端点均连续受限10帧。端点守卫既没有造成该相位，也不能在不提高设备上限的前提下消除它。
3. 不提高1440 counts/s二维设备上限，不新增控制器候选，不修改端点守卫、`EvaluateGate`、shadow或active。现有485个四端点共同域自然保持零损失，P0-5继续`HOLD_SHADOW`。
4. 下一主线从控制器调参转向真实重捕获包络：采集left/right目标重新进入检测区时的首帧水平残差、有效帧间隔和连续限速时长，验证跨域约303～305 counts残差是否代表实机场景。只有实测证明设备存在更高且安全的恢复期输出包络，才评估严格限于恢复窗口的离线候选，并仍须保持485共同域零损失；否则把该动态域认定为当前检测覆盖与物理执行上限共同决定的不可达域。

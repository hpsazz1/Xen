# 分轴主动Profile标定协议

## 背景

既有孤立脉冲确认当前KMBOX_NET、DML、NDI链路约为0.5 px/count，t50约14 ms、t90约15 ms；被动Profile多轮结果却出现X轴0.206～0.526 px/count、Y轴0.318～0.508 px/count和双轴有效窗口不重合。继续累加闭环被动样本无法区分目标运动、手动输入、控制器反馈和真实设备响应，因此本轮不调参，而是建立严格的分轴主动协议。

目标是形成可重复、可拒绝、可审计的机器标定候选。即使协议通过，也不自动写入Profile；`profile_calibration_enabled=false`、r64、shadow和全部运行参数保持不变。

## 采集设计

沿用`xen_physical_response_probe`的安全边界：配置必须是绝对路径，NDI源必须精确指定，设备运动必须显式确认，默认拒绝WIN32，KMBOX冷启动先执行最小正负预热，NDI必须达到原始240 Hz且全程零丢帧。目标和角色保持静止，ROI选择固定高对比纹理。

正式矩阵为：

- 3个独立轮次；每轮重新启动探针，保留独立目录身份；
- 16、32、64 counts三个幅值；8 counts低于当前1 px可靠跟踪门槛，不纳入拟合；
- 每个幅值按X+、X-、Y+、Y-采集，每方向默认10次，硬门禁最低5次；
- 每个trial包含至少300 ms基线、500 ms响应尾部和500 ms隔离间隔；
- 正负命令成对，使视角回到近似原点，禁止斜向或连续闭环激励混入。

探针原summary只记录主轴最终位移。本轮增加`OrthogonalDisplacementPx`、`PixelsPerCount`和`CrossAxisLeakagePercent`，由同一trial的主轴/正交轴基线与尾部中位数计算。主轴仍使用单调包络提取t10/t50/t90/t99，正交轴只用于串扰门禁，不参与响应时间拟合。

## 硬门禁

`tools/analyze_active_profile_protocol.ps1`递归读取`counts<幅值>_run<轮次>/probe_summary.csv`，拒绝以下任一情况：

1. 后端、Git提交或控制器修订身份不唯一；存在无效trial、目录幅值与行内counts不一致，或X/Y任一方向样本不足；
2. 主轴位移方向未与输入counts相反，响应时序次序无效，t90超过100 ms，或任一轮/幅值/轴的中位比例不在0.25～0.75 px/count；
3. 同一轮、幅值、轴的正负方向中位比例差超过两者均值的15%；
4. 同一轮、同一轴在16/32/64 counts间的比例跨度超过中位数15%；
5. 同一轴、同一幅值跨三轮比例跨度超过中位数15%；
6. 同一轴、同一幅值跨三轮t50或t90中位数跨度超过5 ms；
7. 任一轮/幅值/轴的正交泄漏P95超过主轴位移10%。

这些阈值服务于“是否值得人工复核”，不是运行参数。5 ms时序跨度略高于240 Hz单帧4.17 ms量化；15%比例边界远小于既有被动X轴跨轮离散度，同时允许整数像素模板量化。禁止在看到实测后放宽单项门禁或删除失败轮次。

## 输出与安全语义

分析器输出逐轮/幅值/轴CSV和decision。通过时`Recommendation=MANUAL_REVIEW_ONLY`，失败时为`HOLD_DIAGNOSTIC`；两者都固定`ProfileAutoWrite=0`。sweep脚本默认自动执行分析，也可在诊断采集中显式`-SkipAnalysis`，但跳过分析的数据没有Profile资格。

自动测试构造3轮、180个稳定trial验证通过路径，再注入X正方向比例偏差验证方向不对称必然拒绝。C++测试同时验证正交尾部位移不会在响应分位计算中丢失。CTest因此由10项增加为11项。

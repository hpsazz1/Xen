# 故障排查顺序

当出现问题时，请按以下顺序排查：

1. 确认构建类型：DML 或 CUDA。
2. 确认模型类型：DML 使用 `.onnx`，TensorRT 使用 `.engine`，或者如果你希望 CUDA 生成新 engine 则使用 `.onnx`。
3. 确认采集显示的是预期内容。
4. 确认所选 `input_method` 具有其所需的运行环境/设备。
5. 开启有用的日志或诊断信息。
6. 通过匹配的包装器重新构建，并在源码更改后运行真实的 DML 或 CUDA 冒烟测试。

## 检测正常，但瞄准不移动

常见症状：

- 检测正常，但瞄准在游戏中不移动。
- ESP 或预览显示目标，但瞄准和自动射击不工作。
- 自动射击有时工作，但自动瞄准不工作。
- GUI 可以通过 `Home` 键打开，但游戏仍然忽略移动。
- 用户尝试了 `WIN32`、`GHUB`、`MAKCU` 或 KMBOX，但不知道哪个环节失败了。

如果框或预览检测可见，通常采集和模型加载已在正常工作。接下来检查输入链。

`WIN32` 是标准的 Windows 输入路径。某些游戏会忽略或阻止 Win32 合成鼠标事件，因此 `WIN32` 可以在桌面上工作，但在游戏内部却毫无作用。发生这种情况时，更改模型置信度、FOV 或平滑参数不会修复移动问题。

请按以下顺序排查：

1. 检查控制台中关于所选方法的行，例如 `[Mouse] Using WIN32 input.`
2. 如果选择了 `WIN32`，在游戏外测试移动。
3. 如果移动在游戏外正常但在游戏中不正常，请使用其他 `input_method`。
4. 对于屏蔽 Win32 输入的游戏，连接并配置支持的外部输入设备或运行环境，例如 KMBOX、MAKCU、G HUB 或 Razer。
5. 切换方法后，确认设备特定的日志显示已连接/打开状态。当硬件/运行环境方法失败时，应用不会静默回退到 `WIN32`。

有关设置详情和方法特定的检查，请参阅[输入方法](input-methods.md)。

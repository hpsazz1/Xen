# 手动控制权仲裁实现（20260721）

## 背景

按住自动锁定时继续移动物理鼠标，旧链路只看到延迟画面误差和自身自动命令，无法知道用户正在主动修正。自动控制器会在手动修正之后继续执行旧请求，随后对新的误差反向修正，表现为准星持续左右或上下晃动。

## 设计

本轮采用人手优先的共享控制：物理鼠标位移已经直接进入游戏，程序只缩放自动辅助输出，不把人手位移再次入队。Raw Input 按设备包收集物理相对位移，并在自动命令成功发送后登记精确向量；发送后25 ms内完全相同的包归类为自身回注，不作为手动意图。输入按角度换算为度/秒，避免 DPI 和设备计数差异改变阈值。

仲裁器包含四态：`auto`、`blend`、`manual_override`、`recover`。进入阈值持续15 ms，退出阈值为进入阈值的一半且保持80 ms；手动速度达到融合稳定阈值的两倍，或与自动方向余弦不高于-0.25时，立即进入完全接管。融合状态按方向保留自动权重：同向默认0.50，侧向默认0.20，反向为0；松手后用150 ms smoothstep恢复，避免权重跳变。

检测到手动接管时清空待发送队列、重置基础控制器和整数余量，并冻结积分；Raw Input 的物理位移按当前控制消费时刻聚合写入正式自运动补偿时间线，保持历史单调并避免相机延迟把手动转向误判为目标运动。

## 实现范围

- `Xen/runtime/raw_mouse_input.*`：Raw Input 注册、包聚合、自身回注过滤和可测试注入入口。
- `Xen/runtime/manual_control_arbiter.h`：状态机、方向余弦、动态自动权重和恢复曲线。
- `Xen/mouse/mouse.cpp`：输入接入、控制器清理、权重应用、自运动补偿和自动命令登记。
- `Xen/debug/pipeline_tracer.*`：新增原始输入、过滤输入、手动角速度、方向余弦、状态和最终权重字段。
- 桌面 UI 位于“瞄准 → 移动响应 → 人手控制权仲裁”，配置默认关闭；Web 控制台暂不开放该开关，避免远程误启用未经设备归因验证的功能。
- 控制器修订号更新为r66，并由`BuildIdentity::controllerRevision()`统一提供给UI标题和流水线CSV，使构建产物共同标识本次运行行为变更；旧机器Profile缓存按既有严格键安全失效。

## 验证

核心测试覆盖同向融合、反向完全接管、松手恢复以及自动/物理 Raw Input 包分离。功能默认关闭，因此未启用配置时既有输出路径不改变。正式启用前必须采集包含静止、同向、反向和松手恢复的 CSV，确认 `ManualSelfSuppressedCounts*` 与自动发送命令一致，且 `ManualInputCounts*` 不被自动回注污染。

提交前验证结果：`git diff --check`通过；DML与CUDA的`xen.algorithm.basic_tracking`、`xen.config.auto_derive_preserves_movement`、`xen.ui.prediction_visibility`均为3/3通过；DML Release与CUDA Release主程序均成功链接。最终交付构建以本文件所在程序提交为身份来源，后续纯文档提交不使该构建失效。

## 后续建议

先在 DML shadow/legacy 兼容环境完成设备级包归因和阈值复核，再考虑把默认值从关闭改为开启。若驱动把自动命令拆成多个 Raw Input 包，当前精确向量过滤会保守地放弃归因，应先扩展分包协议而不是放宽匹配窗口。

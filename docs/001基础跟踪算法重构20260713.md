# 基础跟踪算法重构（2026-07-13）

## 背景

原移动链路同时包含多目标 Kalman、单目标滑行、外部速度注入、未来位置外推、速度倍率曲线、PurePursuit、输出 EMA、Bezier 和 Wind Mouse。多个模块重复处理速度和平滑，导致参数含义不单一，也无法通过流水线数据准确判断问题属于观测、控制还是轨迹整形。

典型现象是 `minSpeedMultiplier/maxSpeedMultiplier` 从 `0.1` 调到 `0.2` 后输出直接翻倍，而目标速度突刺又会同步提高控制器上限。静止目标因此可能出现越过、反向修正和近中心抖动。

## 原因

1. “速度倍率”同时叠加距离曲线、FOV/灵敏度换算、FPS 缩放和控制器限幅，没有单一物理意义。
2. SOT/Kalman 状态被再次做速度 EMA 和未来外推，形成重复预测。
3. 头身合并计算出的瞄准枢轴没有进入旧 SOT 最终输出，最终使用的是跟踪框中心。
4. 轨迹模拟和输出平滑混在基础控制链路内，基础算法测试会被后处理结果污染。
5. 旧流水线 CSV 同时记录预测、速度倍率和多层平滑，无法独立验证基础移动。

## 排查过程

1. 分别采集左、上、右静止目标的流水线 CSV。
2. 对比 `RawPivot`、预测位置、速度、控制上限、最终计数和队列深度。
3. 确认队列没有积压，异常来自静止目标被估计为数百至上千 px/s。
4. 暂时修正速度滤波和近中心单计数抑制，验证静止目标能够稳定停止。
5. 进一步审计完整链路，确认旧设计存在多层职责重叠，因此决定重写基础层，而不是继续追加补丁。

## 解决方案

当前基础链路固定为：

```text
检测观测
→ 当前帧目标关联
→ 自适应观测滤波
→ 中心误差
→ 帧率无关基础控制器
→ 设备计数发送
```

### 当前帧目标关联

`BasicTargetTracker` 只关联当前检测结果：

- 保留头身合并后的真实瞄准枢轴；
- 已锁定时按距离门限匹配上一观测；
- 无锁定时选择靠近准星且置信度更高的候选；
- 不做速度预测、丢失滑行或纯预测移动。

### 自适应观测滤波

`BasicTargetFilter` 使用时间校正的自适应低通：

- 静止或低速时使用约 30 ms 时间常数抑制检测量化；
- 运动速度升高时将时间常数连续降低到约 6 ms；
- 亚像素残差连续衰减，不使用会产生跳变的硬坐标死区；
- 大幅位置跳变视为新目标并立即重置，不跨目标拖拽状态；
- 输出永远不超过当前观测，不做未来位置外推。

### 基础移动控制

`BasicAimController` 使用一阶时间响应：

- `move_response_ms` 表示误差响应时间，默认 `80 ms`；
- `move_max_speed_cps` 表示设备最大速度，默认 `240 counts/s`；
- 每帧限速由 `counts/s × dt` 得到，因此 FPS 改变不会改变每秒总速度；
- 稳定半径和释放半径按检测分辨率自动推导，并使用滞回避免中心附近反复启停。

### 暂时移除的模块

以下模块不参与当前活动链路、侧栏和配置保存：

- 未来位置预测及检测延迟外推；
- SOT 丢失滑行和外部速度前馈；
- 最小/最大速度倍率及三区间速度曲线；
- PurePursuit；
- Bezier、Wind Mouse、输出 EMA 和过冲整形。

这些模块只有在基础静止与移动目标测试通过后，才能以独立、可开关、可测试的后处理模块重新接入。

## 流水线调整

新版 CSV 只记录基础链路：

- 原始观测：`RawPivotX/Y`
- 基础滤波：`FilteredX/Y`、`ObservedSpeed`、`FilterResidual`
- 控制误差：`ErrorX/Y`、`ErrorDistance`
- 控制请求：`RequestedPixelX/Y`、`RequestedCountsX/Y`
- 限制与状态：`ResponseSeconds`、`MaxCountsPerSecond`、`FrameCountLimit`、`Settled`
- 发送请求：`FinalMx/My`、`QueuedMoveCount`

新版与旧版 CSV 字段不同，禁止混合统计。

## 建议

下一轮只测试基础链路：左、上、右静止目标，以及横向匀速移动目标。确认收敛、稳定和跟随速度后，再单独设计预测模块；轨迹模拟必须最后接入，且不能改变基础控制器的稳定性。

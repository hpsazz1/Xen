# r61有限相机响应影子实现

## 背景与目标

r60九点static实测中，固定60/10/20/30 ms延迟的运行态机动误启用分别为211/86/28/163帧。20 ms是单一阶跃最优点但仍未通过，10和30 ms形成两侧反证，说明真实画面响应跨多个检测帧渐进出现。本阶段不调整常加速度模型、门槛或控制器，只把DML shadow的应用相机模型从固定阶跃扩展为有界线性响应。

候选冻结为响应中心20 ms、完整宽度20 ms，即每条成功发送的命令从发送后10 ms开始、到30 ms完整反映。该区间完全来自10/20/30 ms实测，不新增参数扫描；active仍不可用，正式设备输出继续由legacy产生，CUDA强制保持0宽度阶跃和常速度Kalman。

## 调用链与实现

`AppliedViewMotionModel`继续只接收底层设备确认成功发送的counts、当前Profile换算的`degrees/count`和真实发送时刻。每条命令保存角度增量、响应开始与结束时刻；查询观测时刻或控制时刻时，已完成命令全部累计，正在响应的命令按时间比例线性累计，尚未开始的命令不计入。

响应语义与仓库已有`ViewMotionHistory`一致：delay表示响应中心，response表示完整宽度，开始时刻为`max(0, center-response/2)`。宽度0时开始和结束相同，在中心时刻一次性应用，精确兼容r60和旧配置。历史裁剪会把已完成命令折入稳定基线，不能因512条上限或2秒生命周期使世界角坐标归零。

新增配置`aim_shadow_command_response_ms`，默认0、范围0~100 ms；DML可显式使用20，CUDA编译即使读取到非零配置也强制传入0。UI“瞄准→基础移动→新控制器影子参数”同时显示响应中心和响应宽度。控制器修订号升级为r61。

## CSV与自动验收

流水线CSV在`CommandToFrameDelayMs`后新增`CommandResponseMs`，记录运行时实际生效值而不是原始配置文本。`-RequireFiniteViewResponse`要求DML shadow逐帧精确为20/20，并与既有身份、时间、命令抑制和估计有限性门槛同时执行。

机动验收把`ManeuverActiveSamples`拆为暂停态和运行态。九点换位时未按追踪键的人工视角移动只单独报告；static运行态任何启用仍直接失败，jump/reverse也必须在运行态至少启用一次。非标准文件名目录的场景集合为空时统一按0求和，不再因严格模式读取缺失`Sum`而异常退出。

## 测试设计

- 配置测试覆盖默认0、保存迁移和100 ms上界；
- 应用相机模型测试覆盖宽度0阶跃兼容、10/20/30 ms的0/50%/100%线性响应、裁剪累计、配置变更清空和并发查询有限性；
- CSV测试覆盖新表头与行列一致；
- PowerShell测试覆盖20/20合法候选、0宽度拒绝、static暂停启用允许、static运行启用拒绝、jump/reverse运行态覆盖及非标准文件名。

## 实机门禁

r61 DML构建使用：

```ini
aim_pipeline_mode = shadow
aim_shadow_command_to_frame_delay_ms = 20.00
aim_shadow_command_response_ms = 20.00
aim_shadow_estimator_mode = maneuver_gated_ca
aim_shadow_ca_jerk_std_dps3 = 8000.00
aim_shadow_maneuver_rate_threshold_dps = 12.00
aim_shadow_maneuver_hold_ms = 120.00
profile_calibration_enabled = false
```

首轮只复测九点static，每点按住约3秒、换位时松开，保存CSV和实际配置。运行态机动启用必须为0，恢复首帧有效，暂停命令、时序、选择计数和有限响应契约均为0违例。通过后才允许使用同一身份采集jump/reverse；失败则保持`HOLD_SHADOW`并依据逐轴残差决定是否进入分轴响应，不回到固定延迟或CA门槛扫描。

# 冻结320与DML性能诊断

## 最终积分结论

2026-07-14 使用最新 `XenMovingPI320` 与 `XenMovingPI280` 数据完成最后一轮复验。六份CSV均为 `BuildRevision=e7d3b860fce9`、`ControllerRevision=4`，两档均保持80 ms响应和1440 counts/s上限。

| 场景与指标 | 320 ms | 280 ms | 280相对结果 |
|---|---:|---:|---|
| 左移末尾1秒P95 | 2.70 px | 2.50 px | 改善7.4% |
| 右移末尾1秒P95 | 2.49 px | 2.48 px | 改善0.4% |
| 高频误差P95 | 18.32 px | 17.14 px | 改善6.4% |
| 高频恢复P95 | 77 ms | 72 ms | 改善6.5% |
| 反转速率 | 2.93次/秒 | 2.72次/秒 | 差异7.5% |

280 ms各项收益均未达到既定8%～10%推广门槛，反转速率差也超过5%协议要求。继续细分300、260等相邻档位的预期收益低于测试噪声和维护成本，因此正式冻结现场移动积分基线为320 ms，源码默认0仍用于未完成现场标定的新部署。

## 下一阶段目标

现有NDI数据长期显示源到达约220～234 FPS，而DML检测发布约88～92 FPS，推理链路已经是主要吞吐限制。下一阶段只分析DML预处理、ONNX Runtime推理、输出交接、后处理和NMS，不接入预测、不调整控制器、不改变采集坐标契约。

## 性能诊断实现

DML检测器原有阶段计时只供覆盖层读取，跨推理线程和UI线程访问没有同步，也没有进入现场CSV。本轮改为一次发布对应一个线程安全快照，并新增：

- `DmlPreprocessMs`
- `DmlInferenceMs`
- `DmlCopyMs`
- `DmlPostprocessMs`
- `DmlNmsMs`
- `DmlTotalMs`

`DmlNmsMs`包含在`DmlPostprocessMs`内，不得重复加入总耗时。DirectML的`session.Run`是同步调用，GPU执行、同步和ORT内部输出准备主要计入`DmlInferenceMs`；`DmlCopyMs`仅表示ORT返回到后处理开始之间的交接时间，通常接近0。

覆盖层与鼠标流水线均通过同一锁保护快照读取，避免读取到不同推理帧混合或发生数据竞争。

## 分析工具

新增正式脚本：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File tools/analyze_dml_performance.ps1 `
  -DataRoot C:\Users\User\Desktop\XenDmlPerformance `
  -OutputCsv C:\Users\User\Desktop\XenDmlPerformance\dml_performance_summary.csv
```

脚本按文件和总体输出各阶段均值/P95、推理占比、理论流水线FPS、实际发布FPS、源到达FPS、观测年龄P95和构建身份。旧CSV缺少阶段字段时只警告并跳过，不与新性能数据混用。

## 下一轮性能测试方案

1. 使用本轮新DML干净构建，保持检测分辨率320、完整源2560×1440、NDI、80/320/1440控制配置和同一模型。
2. 流水线缓冲至少设置3000帧，分别采集静止中心目标30秒、持续水平移动30秒、高频反转30秒；每份CSV至少1000个有效`DmlTotalMs>0`样本。
3. 每份CSV先核对`BuildBackend=DML`、同一`BuildRevision`、`ControllerRevision=4`和六个DML耗时字段。
4. 使用性能脚本导出总体与分场景汇总。若`InferenceSharePct>=70%`，优先分析模型输入、ORT图优化和DirectML执行；若预处理或后处理平均占比达到15%以上，才进入对应CPU路径优化。
5. 任一优化必须使目标阶段均值或P95至少改善10%，并使`PublishedInferenceFps`至少提高8%；`ObservationAgeP95Ms`不得回退超过10%，移动目标稳态与反转指标不得回退超过10%。
6. DML+NDI取得明确收益后，再用Desktop Duplication或WinRT复测相同模型，区分推理优化与采集链路收益；不得同时修改模型、分辨率和采集方式。
7. DML性能闭环完成后才评审预测补偿模块。

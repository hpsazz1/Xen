# 机器Profile响应覆盖阻断与解耦

## 背景

人工候选已经证明严格键可在实机精确命中 Level 3，并在 `aimMode` 变化后立即降至 Level 2。下一步使用相同构建 `DML|228aa1d7f6b8|r64`、相同 NDI/设备/游戏配置，对 static、horizontal_jump、horizontal_reverse 各采集三轮 L3 与 L2 shadow 数据。两组均保持命令和轨迹命令抑制，前馈增益与 shadow 积分均为零，因此可以隔离机器缓存响应模型对估计链的影响。

## 设计修复

主动脉冲的 t50/t90 只证明单次阶跃时序，不足以证明“t50 中心、零响应宽度”的连续自运动模型。修复后：

- schema v1 继续保存 X/Y t50/t90，Level 3 决策继续暴露证据推导的 14.096/0 ms 供审计；
- `calibratedViewResponseEnabled` 固定为 false，精确命中不再等价于响应应用资格；
- 新增 `selectMachineProfileViewResponse` 作为唯一选择点。未获独立资格时，无论 Level 3 是否命中，都明确恢复配置响应；DML 为 20/20 ms，CUDA继续按既有规则使用零宽度；
- 每次配置刷新都会重新执行选择，避免旧 L3 的 14/0 ms 状态残留；
- UI 增加“主动标定响应应用”及证据值，流水线 CSV 增加 `MachineProfileCalibratedResponseEnabled`，候选生成器输出 `CalibratedResponseEnabled=0`。

该改动不改变 BasicAimController、legacy参数、设备队列、主动协议或缓存键，所以控制器修订保持 r64；新的 Git 构建身份用于区分修复前后程序。active继续冻结。

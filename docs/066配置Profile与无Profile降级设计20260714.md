# 配置Profile与无Profile降级设计

## 背景

用户确认CS2的武器移动速度、行走和下蹲比例只作为运动量级参考，同时提出两个关键问题：游戏profile由何处获取、不同游戏profile缺失或不一致时怎么办。

现有Xen已经存在程序内部游戏profile。`Config::GameProfile`保存在`config.ini`的`[Games]`段，当前字段为：

```text
GameName = sens, yaw, pitch, fovScaled, baseFOV
```

默认包含`CS`与`UNIFIED`，`active_game`不存在时回退到其中之一。该结构目前只解决“游戏角度如何转换为鼠标counts”，不是完整的自动标定、运动模型或跨游戏数据库。

本轮目标是明确哪些内容继续属于用户配置，哪些应由程序自动测量，哪些只能作为可丢弃先验，并设计无profile时仍能安全运行的降级链路。

## 结论先行

不存在成熟、统一、长期可靠的全游戏运动profile数据库。游戏更新、武器、姿态、FOV模式、输入系统和移动机制都会改变数据，互联网资料只能作为初始参考。

Xen不应依赖外部网站或在线服务启动。推荐使用三层数据：

1. 用户配置profile：稳定、可读、可手工修改的游戏坐标初值。
2. 机器标定缓存：程序在当前设备、采集链路和分辨率下自动测得的实际响应。
3. 会话运动统计：本次运行从相对视线观测中学习，只用于Q/R、置信度和诊断。

任何层缺失都允许运行；任何层与真实观测冲突时，优先相信当前可验证观测并降低或清除先验权重。

## 第一层：现有config游戏profile

### 保留的职责

当前`GameProfile`可以继续作为用户配置入口，提供：

- 游戏或瞄准模式名称
- 灵敏度`sens`
- 水平`yaw`
- 垂直`pitch`
- 是否使用FOV缩放
- 缩放基准FOV

通过这些字段可推导：

```text
horizontalDegreesPerCount = sens × yaw × fovScale
verticalDegreesPerCount = sens × pitch × fovScale
```

Mouse Sensitivity网站、游戏配置文件、控制台测量或用户手工数据都只能用于填写这一层的初始值，程序运行时不需要访问网站。

### 建议后续增加的稳定字段

在保持旧格式可读取的前提下，后续可增加：

- 双精度实际HFOV/VFOV
- 瞄准模式标识，例如Hipfire、ADS、Scope1、Scope2
- profile数据来源：默认、用户输入、网站参考、程序实测
- profile修订号和最后更新时间
- 是否允许自动标定覆盖派生值

这些字段描述用户期望和游戏坐标语义，适合保存在`config.ini`。不同开镜模式的FOV或灵敏度变化应使用不同profile，而不是在同一估计器状态中热改参数。

## 第二层：机器标定缓存

自动学习结果不应直接覆盖用户维护的`config.ini`。建议使用独立、可删除、可重建的标定缓存，按以下组合建立键：

```text
游戏/瞄准模式
+ 完整源分辨率和ROI
+ 捕获后端
+ 推理后端
+ 输入设备类型
+ 灵敏度与FOV配置摘要
```

缓存内容包括：

- 实测中心`countsPerPixelX/Y`或`pixelsPerCountX/Y`
- 由实测响应反推的有效`degreesPerCountX/Y`
- command-to-frame延迟P50/P95
- 可选的一阶响应时间
- 标定样本数、残差、可信度和时间
- 标定时的源尺寸、FOV、后端与设备身份

该缓存属于当前机器和链路，不应当作游戏通用数据，也不应提交进仓库。用户更换显示模式、OBS裁剪、NDI链路、输入设备、FOV或灵敏度后，缓存键变化并自动失效。

### 自动标定来源

程序可按成熟程度分两步获得：

1. 被动标定：利用正常控制期间已发送counts和后续画面反向位移，通过时间对齐和稳健回归估算比例与延迟。
2. 主动标定：在明确测试模式下发送小幅、对称、限速的角度脉冲，使用静止目标或稳定场景拟合响应。

被动标定无额外操作但容易受目标移动污染；主动标定更可靠但必须由用户明确进入测试模式。首版应先离线使用已有static CSV验证，不立即在实机自动发送标定动作。

## 第三层：会话运动统计

武器速度表、走跑蹲比例和不同游戏移动上限不进入运行硬配置。程序直接从当前检测序列统计：

- 相对视线角速度分布
- 角加速度或innovation分布
- 检测测量噪声R
- 常规运动所需过程噪声Q
- static、持续移动、停止和反转的NIS范围
- 前馈生效时的误差与方向稳定性

这些数据首先只存在内存和CSV。只有后续证明跨会话稳定，才考虑保存成带样本数和可信度的软先验；它必须可以随时清除，不能覆盖用户坐标配置。

## CS2世界速度数据的正确用途

刀250 u/s、步枪约215~225 u/s、AWP约200 u/s、行走和下蹲比例等数据可以提供以下参考：

- 离线合成目标与角色同向、等速、速度差和反向叠加场景。
- 检查相对速度`Vtarget−Vself`的符号和量级是否合理。
- 设计低速、中速、高速与突然切换的覆盖矩阵。
- 在未来获得可靠三维遥测时作为游戏适配器初值。

它们不能直接用于：

- 从u/s换算固定px/s或deg/s，因为缺少目标深度和相机方向。
- 作为Kalman拒绝观测或撤销预测的硬阈值。
- 推断当前玩家实际速度，因为还受武器、Shift、下蹲、加速度、碰撞、斜坡、空中和游戏更新影响。
- 迁移到其他游戏作为统一速度标准。

跨游戏测试只提取“慢速、常速、高速、加速、减速、停止、反转”等抽象运动类别，不保留CS2武器名称。

## 无profile降级状态机

### Level 3：可信用户配置和可信标定缓存

- 使用HFOV/VFOV与`degreesPerCount`进入完整角度Kalman。
- 使用标定后的command-to-frame模型补偿相机旋转。
- 启用相对视线角速度前馈。

### Level 2：有用户配置但无标定缓存

- 使用配置推导角度与counts关系。
- command-to-frame使用保守初值并只记录影子诊断。
- 降低前馈增益上限，后台积累被动标定样本。

### Level 1：缺少可信FOV或游戏profile

- 使用完整源尺寸归一化坐标，例如`2×pixelDelta/sourceSpan`。
- 根据自身已发送counts和画面位移学习中心局部`px/count`。
- 状态估计仍可在归一化图像空间运行，但不宣称真实角度。
- 在标定置信度建立前关闭或严格限制速度前馈。

### Level 0：配置与自动标定均不可靠

- 使用低增益、限速、带框内死区的直接追踪。
- 关闭预测、速度前馈、积分保持和目标丢失coast。
- 保留目标丢失立即停止和身份切换重置。
- UI明确显示“未标定安全模式”，不能静默套用CS或其他游戏profile。

## profile失配检测

不能只根据profile名称判断有效。以下任一情况持续出现时应降低可信度：

- 当前源尺寸、ROI、FOV或后端身份与缓存键不同。
- 已发送counts与观测到的画面位移比例长期偏离标定范围。
- command-to-frame残差或NIS持续异常。
- 用户切换ADS、Scope、分辨率、灵敏度或FOV。
- 水平/垂直响应比例与配置不符。
- 标定样本过旧、样本数不足或可信度低。

处理顺序为：

1. 停止使用运动软先验。
2. 降低或关闭角速度前馈。
3. 重置状态估计器与相机响应历史。
4. 进入Level 2或Level 1重新标定。
5. 严重失配时进入Level 0安全直追。

绝不能自动切换到名称相近的其他游戏profile。

## 与成熟方案的关系

成熟的是以下工程原则，而不是一个现成全游戏profile库：

- 配置与自动标定分离
- 系统辨识估算输入到输出的比例和延迟
- Kalman使用Q/R与协方差表达不确定性
- innovation/NIS检测模型或测量失配
- 缓存按设备与配置身份失效
- 低置信度时功能降级而不是继续强预测

这些方法在控制、机器人和视觉融合系统中成熟可用；Xen仍需要结合自身CSV、输入设备和捕获链路实现并验证。

## 推荐迁移方式

1. 保持现有`[Games]`格式和`CS/UNIFIED`回退可读取，避免破坏已有配置。
2. 先增加只读的“有效坐标参数”诊断，显示最终`degreesPerCount`、FOV、来源和可信度。
3. 再增加独立标定缓存和失效键，不自动覆盖用户配置。
4. 完成static离线比例与延迟标定后，才启用Level 2到Level 3切换。
5. 最后实现Level 1归一化坐标学习和Level 0安全直追。

在上述降级链路完整前，不应新增互联网自动下载profile功能。

## 本轮结论

用户理解正确：现有`config.ini [Games]`就是profile的基础，但它只应保存稳定的用户游戏坐标参数。自动测得的设备、OBS/NDI和输入响应应进入独立机器缓存；CS2移动速度等数据只用于离线场景参考；运行中的运动分布由程序从相对视线观测自适应学习。无profile、profile不一致或缓存过期时，系统必须逐级关闭前馈并退化到安全直追，绝不能强制套用CS或外部数据库参数。

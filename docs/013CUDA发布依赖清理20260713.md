# CUDA 发布依赖清理

## 背景

`build\cuda\Release` 曾混入多次构建和手工复制留下的 DLL。目录中同时存在 CUDA、cuDNN、TensorRT、DirectML 风格依赖、测试程序和链接中间文件。直接把整个目录复制到另一台电脑，既会显著增大发布体积，也无法判断实际加载的是当前依赖还是旧文件。

本次目标不是手工清空一次目录，而是让每次 Release 构建都生成可重复验证的运行时闭包。

## 排查过程

使用 Visual Studio `dumpbin /dependents` 检查了 `Xen.exe`、`opencv_world4130.dll` 以及第一层 CUDA/NPP/MSVC 依赖：

- `Xen.exe` 直接依赖 OpenCV、TensorRT 核心、ONNX Parser 和 NDI，不依赖 ONNX Runtime、DirectML 或 cuDNN。
- OpenCV CUDA 直接使用 cuBLAS、cuFFT 和部分 NPP 模块。
- `cublas64_13.dll` 继续依赖 `cublasLt64_13.dll`；所用 NPP 模块继续依赖 `nppc64_13.dll`。
- OpenCV CUDA 使用动态 VC143 CRT，因此即使 `Xen.exe` 自身采用静态 CRT，发布目录仍需保留四个实测需要的 MSVC DLL。
- 当前 OpenCV 构建已禁用 cuDNN，TensorRT 核心也没有静态 PE 导入 cuDNN、NVRTC、NVJitLink 或插件 DLL。

旧目录中的问题分为三类：

1. 旧运行库：`cudnn*`、`nvrtc*`、`nvJitLink*`、未使用的 NPP 模块和多余 MSVC CRT。
2. 未启用能力：`nvinfer_plugin*`、`nvinfer_lean*`、`nvinfer_dispatch*`、`cpbox_mouse_dll.dll`。
3. 非发布文件：`Xen.exp`、`Xen.lib` 和 `xen_*_tests.exe`。

## 设计与实现

构建系统现在采用以下闭环：

- 从当前 CUDA Toolkit 明确复制 OpenCV 实际需要的 cuBLAS、cuFFT 和 NPP DLL。
- 从当前 Visual Studio VC143 Redistributable 明确复制 OpenCV 实际需要的 CRT DLL。
- 继续从当前 TensorRT SDK 复制核心、ONNX Parser 和 builder resource DLL。
- 构建结束后按禁止清单删除旧后端、旧插件和非运行时文件。
- 单元测试程序输出到 `build\<backend>\tests\Release`，不再污染应用发布目录。
- 导入库输出到 `build\<backend>\lib\Release`。
- 新增发布布局测试，分别校验 CUDA/DML 必需文件和禁止文件。

TensorRT 的 `nvinfer_builder_resource_*.dll` 体积较大，但仍保留。它们用于在目标电脑上从 ONNX 构建 TensorRT engine；删除后只能分发预先生成且与目标 TensorRT/显卡环境兼容的 engine，不符合当前程序支持现场建引擎的完整流程。

## CUDA Release 保留原则

除了 `Xen.exe`、配置、模型、NDI 和鼠标接口 DLL，CUDA 发布目录保留：

- `opencv_world4130.dll`
- `nvinfer_10.dll`、`nvonnxparser_10.dll`
- `nvinfer_builder_resource_*.dll`
- `cublas64_13.dll`、`cublasLt64_13.dll`、`cufft64_12.dll`
- `nppc64_13.dll`、`nppial64_13.dll`、`nppicc64_13.dll`、`nppidei64_13.dll`、`nppig64_13.dll`、`nppist64_13.dll`、`nppitc64_13.dll`
- `concrt140.dll`、`msvcp140.dll`、`vcruntime140.dll`、`vcruntime140_1.dll`

显卡驱动提供的 `nvcuda.dll` 不应从开发机复制，必须由目标电脑安装的 NVIDIA 驱动提供。

## 验证方式

最低验证流程：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\tests\build_script_tests.ps1
cmake --build build\dml --config Release
ctest --test-dir build\dml -C Release --output-on-failure
cmake --build build\cuda --config Release
ctest --test-dir build\cuda -C Release --output-on-failure
git diff --check
```

发布时应复制整个对应后端的 `Release` 运行目录以及实际使用的配置和模型，不要从 DML/CUDA 两个目录交叉挑选 DLL。

## 后续建议

- CUDA Toolkit、TensorRT 或 OpenCV 升级后，必须重新检查 PE 依赖闭包并同步发布布局测试，不能只改 DLL 文件名。
- 若未来提供“仅运行预编译 engine”的轻量包，应作为明确的独立发布模式，并在启动时禁止 ONNX 现场构建；不能靠人工删除 builder resource 实现。

# 统一采集帧率诊断与最新帧邮箱

## 背景

2026-07-13 的 NDI 静止目标复测包含左、上、右三组共 629 条流水线记录。旧版已经能区分 NDI 声明、NDI 接收、捕获处理和检测发布 FPS，但其他采集后端没有同等判断；同时 `NdiDroppedFrames=0` 与接收约 240 FPS、消费约 141～153 FPS 的事实矛盾。

## 数据结论

| 数据 | NDI 声明 FPS | NDI 实际接收 FPS | 捕获处理 FPS | 检测发布 FPS | 观测年龄 P95 | 首次稳定 |
|---|---:|---:|---:|---:|---:|---:|
| 上 | 240.0 | 237.3 | 145.9 | 101.2 | 18 ms | 368 ms |
| 右 | 240.0 | 236.8 | 141.1 | 95.6 | 18 ms | 405 ms |
| 左 | 240.0 | 238.5 | 153.3 | 99.0 | 18 ms | 400 ms |

发送声明与实际接收差距不足 1.4%，说明 OBS NDI、局域网和 NDI 解码没有形成主要瓶颈。捕获处理低于输入约 84～96 FPS，检测发布又低于捕获处理约 40～54 FPS，当前性能上限主要位于辅机消费和 DML 推理。移动请求队列最大深度始终为 1，暂时没有依据继续提高控制器速度；盲目提高只会增加中心越过风险。

## 原因

NDI 和 UDP 原实现使用小队列，但消费端每次直接取最新帧并静默清空其余帧。只有队列达到容量上限时才增加丢帧计数，因此高帧率输入被正常淘汰时仍显示为 0。这不影响最新帧低延迟策略，却会误导性能判断。

不同后端过去还使用不同统计口径：NDI 有协议声明和接收线程速率，UDP 只有累计计数，本地桌面与摄像头只有捕获循环 FPS，无法横向比较。

## 设计与实现

1. 在 `IScreenCapture` 建立统一输入源诊断契约：声明 FPS、真实到达 FPS、累计输入、累计淘汰、编码尺寸。
2. NDI 与 UDP 使用容量为一的最新帧邮箱。新帧替换尚未消费的旧帧时立即计数，避免排队处理过期画面。
3. Desktop Duplication 根据真实成功获取事件统计输入 FPS，并将 `AccumulatedFrames-1` 计为采集 API 报告的遗漏帧。
4. WinRT 根据成功返回帧统计输入 FPS；API 不提供可靠声明值，因此显示 `N/A`。
5. 虚拟摄像头读取设备打开后实际报告的 `CAP_PROP_FPS` 作为声明值，并单独统计真实取得速度。
6. UDP MJPEG 没有发送端帧率元数据，因此声明值显示 `N/A`，不使用配置上限冒充源声明值。
7. UI 和 CSV 对全部后端统一显示四级速率：输入源声明、输入源实际到达、捕获处理、检测发布。
8. CSV 新增 `SourceDeclaredFPS/SourceReceiveFPS/SourceReceivedFrames/SourceDroppedFrames`；原 `Ndi*` 字段暂时保留，避免破坏已有分析脚本。

## 诊断口径

- 声明值明显低于期望：检查发送软件、设备模式或驱动协商。
- 实际到达明显低于声明：检查网络、解码或采集 API。
- 实际到达高于捕获处理且源帧淘汰增加：接收端主动保留最新帧，说明消费速度低于输入速度；这是低延迟反压，不等同于网络丢包。
- 捕获处理明显高于检测发布：推理、预处理或后处理成为主要瓶颈。
- 本地桌面源没有协议声明帧率时必须显示 `N/A`；显示器刷新率只作硬件参考，不代替真实到达统计。

## 验证

- 最新帧邮箱单元测试覆盖首次写入、覆盖计数、取回最新值和无隐藏积压。
- 流水线 CSV 表头测试覆盖统一字段和旧 NDI 兼容字段。
- 基础算法与流水线单元测试通过。
- DML Release 完整构建通过。
- CUDA Release 完整构建通过，共享采集接口未破坏 TensorRT/GPU 路径。

## 后续建议

下一轮使用新程序分别复测 NDI、Desktop Duplication 和 WinRT。NDI 预期会出现非零 `SourceDroppedFrames`，其增长率大致对应输入 237～239 FPS 与捕获处理 141～153 FPS 之间的差值。确认统一统计后，再单独用基准工具分析 DML 的预处理、推理和后处理耗时，避免把推理上限误归因于采集链路。

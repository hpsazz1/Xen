# r63 静止回退与 r64 尾部候选

## 背景

物理响应与被动 Profile 标定阶段结束后，计划先复核 r63 的运行态 shadow/static 安全门禁，再决定是否进入分轴主动标定设计。现有三份 `profile_calibration_run1~3.csv` 均来自同一静止目标、角色不移动的真实链路，可直接作为比单轮九点测试更长的 static 重复样本。

三份数据身份统一为 `DML / 52096c1e775f / r63`，使用 shadow、20/20 ms 有限响应、`maneuver_gated_ca + 8000/12/120`、不确定度增益1.25和尾部20 ms。Profile 标定器只读成功命令与后续支点，不写入控制链。

## 排查过程

正式分析器把三份文件按 static 语义隔离后，共检查6099帧。构建身份、模式降级、时序、估计有限性、暂停零命令、暂停连续观测、选择序列、有限响应和逐轴不确定度计算均为0违例。

第1、3轮运行态机动启用均为0；第2轮出现14帧连续启用，文件级失败。这段位于首个运行段恢复后约145 ms：大幅水平追赶命令的建模响应率已经归零，Kalman速率随后反向过冲，门控证据峰值18.53°/s，越过12°/s物理门槛。首次误触只有1帧，之后由120 ms机动保持扩展到14帧；这14帧与legacy的稳定收敛态重合，但设备输出仍由legacy产生，未突破shadow安全隔离。

该结构与r62旧失败相同，说明r63的20 ms不确定度尾部只通过了当时单轮样本，没有覆盖本次更强的真实追赶响应。问题不是手动移动、暂停边界、Profile结果应用或正式设备路径变化。

## 候选设计与离线验证

保持响应中心/宽度20/20 ms、增益1.25、门槛12°/s和保持120 ms不变，只扫描尾部20/25/30/35/40/50 ms。数据集由本次三轮static与既有r63 jump、fast reverse正样本组成。

35 ms是首个使三轮static运行态启用、settled侵入和失败段全部归零的候选。此时jump仍在7/7运行段启用277帧，fast reverse仍在6/6运行段启用845帧；40和50 ms虽也通过负样本，但会继续减少正样本启用帧，因此不采用。

r64仅把DML shadow命令响应速率不确定度尾部从20 ms扩大到35 ms，并同步CSV分析契约；真实相机角度补偿仍为20/20 ms，CUDA仍保持常速度Kalman、0增益和0尾部，active仍强制降级为shadow，正式设备输出不变。

## 后续门禁

离线结果只允许生成r64实机候选，不能直接宣布通过。下一步只采一轮r64九点static：开始前重置流水线一次，点位间仅松键换位；要求所有运行段的机动启用和settled侵入均为0，同时身份、shadow抑制、暂停零命令、时序、35 ms尾部及逐轴计算契约均为0违例。static通过后再决定是否补采jump/reverse，不进入active，也不启动分轴主动Profile标定。

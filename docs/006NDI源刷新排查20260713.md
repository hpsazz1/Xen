# NDI 源刷新排查（2026-07-13）

## 背景与现象

OBS NDI 画面已经能进入捕获预览，配置使用 `ndi_source_name = HPSAZZ (main)`，但设置界面点击“刷新 NDI 源”没有可见反应，下拉列表仍显示未找到源。新采集的左、右、上 CSV 均包含有效画面和控制数据，说明视频接收链路正常，问题集中在 UI 独立发现链路。

## 原因与排查过程

NDI 捕获线程持有一个长期 finder，并按配置名称持续重试连接；UI 的刷新按钮会另建一个临时 finder。旧实现创建临时 finder 后立即调用 `NDIlib_find_get_current_sources`，没有给 mDNS 首轮发现留出时间，因此通常得到空列表。

空结果又把 `ndi_sources_loaded` 设置为 false，下一 UI 帧会自动再次调用刷新。结果是不断创建和销毁 finder，却始终在发现结果返回前读取；按钮没有加载状态，用户看到的表现就是“点击无反应”。NDI SDK 头文件明确说明全局 initialize/destroy 不会对现存对象产生不利影响，因此本次不引入额外全局生命周期管理，避免无依据扩大修改范围。

## 设计与实现

- UI 首次打开和手动刷新都通过单个后台任务执行，禁止并发重复刷新。
- 新 finder 先读取已有缓存；为空时调用 `NDIlib_find_wait_for_sources`，最长等待 1500 ms，再读取结果。
- 搜索期间显示“正在搜索 NDI 源...”和禁用态“正在刷新...”，不阻塞 ImGui 和捕获预览。
- 空结果也视为一次完成，不再每 UI 帧自动重启。
- 发现结果排序并去重，使下拉列表稳定。
- 捕获线程继续独立按完整名称连接，因此 UI 列表和视频是否已有数据不再互相误判。

## 数据验证

本轮三份 CSV 均记录 `Resolution=320`、`SourceWidth=2560`、`SourceHeight=1440`，证明完整 FOV 配置已经生效。左、右主轴误差越零次数均为 0，上方向为 1；相较修复前的 13～19 次，坐标比例导致的严重振荡已经消除。平均处理 FPS 分别约为 130、140、133，观测帧龄平均约为 11.8～13.3 ms，应用移动队列最大深度仍为 1。

## 验证与后续建议

提交前执行 `git diff --check`、基础算法单元测试、DML Release 和 CUDA Release 构建。运行时点击刷新，验收加载状态、1.5 秒内结果回填以及捕获预览不中断；若仍发现不到但指定名称能连接，应检查 mDNS/网卡/防火墙，同时保留完整源名称直连作为稳定回退。

# 角度状态与延迟控制重构细化

## 背景

`docs/063跨游戏预测与控制架构重评20260714.md`已经确定下一代方向为“视线角状态估计 + 时间戳延迟补偿 + 位置反馈和角速度前馈”。本轮继续检索成熟实现，并结合用户从Mouse Sensitivity网站取得的CS2标定数据与现有代码时间轴，细化最小可落地方案。

本轮重点回答四个问题：

1. 首版应使用Alpha-Beta、Kalman、IMM还是更复杂的UKF。
2. 当前static伪速度与反复门控是否只是滤波器问题。
3. OBS NDI、DML推理、鼠标设备和游戏画面延迟应如何分离。
4. 如何把目标预测从“追逐提前点”改为真正的角速度前馈。

## CS2角度与像素标定核对

用户提供的CS2 Hipfire数据为：

- 灵敏度：1.4
- DPI：800
- 分辨率：2560×1440
- 实际HFOV：106.260205°
- 实际VFOV：73.739795°
- 360度距离：37.1104 cm
- 360度测量值：11688.3117 counts
- 画面中心像素比率：0.5161 px/count

CS2当前profile的yaw为0.022°，因此：

```text
每count角度 = 1.4 × 0.022 = 0.0308°
每360° counts = 360 / 0.0308 = 11688.3117
cm/360 = 11688.3117 / 800 × 2.54 = 37.1104
```

106.260205°水平FOV与2560宽度对应针孔焦距约960 px；中心位置每count投影为：

```text
px/count = 960 × tan(0.0308°) = 0.516059
counts/px = 1 / 0.516059 = 1.937763
```

这些结果与网站完全一致，证明项目r10采用的透视换算方向正确。当前配置使用整数`106/74`，相对精确FOV造成的中心换算误差约为水平+0.47%、垂直-0.47%，不足以解释现有震荡和jump问题。

### 对跨游戏配置的意义

下一代坐标适配层应使用以下规范量：

- `horizontalDegreesPerCount`
- `verticalDegreesPerCount`
- 双精度实际HFOV/VFOV
- 完整源尺寸与中心ROI几何

游戏灵敏度、yaw/pitch和FOV缩放可以用于推导每count角度，但估计器不应直接依赖各游戏的灵敏度字段语义。DPI、显示器尺寸和cm/360仅用于用户参考，不进入运行控制数学。开镜、缩放和不同瞄准模式改变FOV或角度倍率时，应切换独立profile并重置估计器。

## 现有时间轴的结构性问题

### 帧时间戳不是源画面生成时间

当前捕获线程在`GetNextFrameCpu()`返回后或GPU帧取得后调用`steady_clock::now()`，随后把该时间作为`frameTimestamp`传给检测器。这个时间能准确覆盖本机预处理、推理、发布到控制的年龄，但不能自动覆盖：

- 游戏渲染与Present等待
- OBS采集和插件处理
- NDI编码、网络传输与解码
- 接收队列中在取帧前已经发生的等待

因此当前`ObservationAgeSec`主要是“本机取得画面之后的年龄”，不是目标像素真正生成到控制时刻的完整端到端年龄。固定额外50 ms前瞻实际上同时承担了未知上游延迟和用户提前偏好的两种职责，换后端或换设备后必然失配。

### 视角补偿假设指令立即出现在画面

当前`recordMotionCompensationStep()`在鼠标设备成功接收counts后立即把对应像素/视角变化写入历史；控制链查询观测时刻与控制时刻的累计视角，并将其用于稳定目标pivot。

真实链路却是：

```text
程序发送counts
→ 设备/网络处理
→ 游戏读取输入
→ 相机姿态更新
→ 游戏渲染
→ OBS/NDI传输
→ 辅机收到画面
```

发送时刻与该变化首次出现在观测画面的时刻之间存在明显延迟，而且可能有短暂响应展开。把counts按发送时刻立即计入视角历史，会在新画面尚未体现该移动时提前补偿；随后真实画面变化到达，又形成反向残差。r11到r14使用保持帧数和重新武装门槛抑制的static伪速度，本质上很可能包含这一时间错位，而不只是检测噪声。

## 继续检索后的算法取舍

### 首版选择线性角度Kalman

相对视线状态直接定义为yaw/pitch角和角速度后，状态转移与测量都是线性的，不需要EKF或UKF：

```text
state = [losX, losRateX, losY, losRateY]
measurement = [losX, losY]
```

每个轴可先独立使用2×2常速度模型：

```text
F = [[1, dt],
     [0,  1]]

H = [1, 0]
```

过程噪声使用白噪声角加速度模型，使常速度滤波器允许短时变速。MathWorks调参资料明确指出：过程噪声越大，滤波器越愿意跟随新机动，也越容易受测量噪声影响；真正重要的是过程噪声与测量噪声的相对比例，并且必须用覆盖全部预期运动类型的数据调参。

Alpha-Beta可作为更小、更易核对的离线基线，但Kalman能够提供状态协方差、测量残差和残差协方差，后续机动识别、异常观测处理和前馈置信度都需要这些量，因此正式候选优先Kalman。

### 不把One Euro叠加在Kalman之前

One Euro适合独立解决“低速去抖、高速少滞后”，但若先对pivot做One Euro，再把结果作为Kalman测量，会形成两个相互耦合的延迟与增益系统，使Q/R难以解释。首轮影子测试应比较：

1. 原始角度 + One Euro；
2. 原始角度 + Alpha-Beta；
3. 原始角度 + Kalman。

不能默认串联One Euro和Kalman。检测框宽高、安全走廊等非状态量仍可单独使用轻量低通。

### 用创新统计替代像素速度门槛

Kalman残差为实际测量与预测测量之差，残差协方差同时包含状态不确定度和测量噪声。Mahalanobis/NIS形式：

```text
innovation = z - Hx
S = HPHᵀ + R
NIS = innovationᵀ S⁻¹ innovation
```

该统计量比固定的“640 px/s”更适合跨FOV、跨帧率和跨游戏判断当前观测是否符合模型。但高NIS不能一律丢弃：

- 检测置信度低、框贴边或外观异常且NIS高：更可能是测量跳点，可提高R或拒绝该次更新。
- 检测置信度高且NIS高：更可能是真实停止、反转或jump，应临时提高Q、降低速度前馈或切换机动模型，不能继续相信旧速度。

首版先记录NIS并离线选择范围，不在运行代码中立刻加入复杂自适应规则。

### IMM、Singer与复杂非线性滤波暂缓

- IMM适合同时运行静止、常速度和高机动模型，但首版先验证单Kalman加过程噪声是否已足够。
- Singer模型用相关加速度描述机动目标，比固定加速度更合理，但需要额外的机动时间常数和加速度方差；在角度状态与时间轴尚未验证前引入会增加不可辨识参数。
- UKF、粒子滤波适合非线性三维状态或非高斯问题；当前直接观测角度的二维控制不需要这些成本。

## 相机控制感知的状态估计

### 独立维护相机视角状态

不要再把目标pivot先加上“发送即生效”的累计像素。应维护一个带时间响应的相机角度估计。这里仅建模相机yaw/pitch旋转，不包含角色WASD平移：

```text
cameraAngle(t) = 历史鼠标命令经过响应延迟模型后的累计角度
```

在观测时刻：

```text
rotationStabilizedLosAngle = measuredRelativeAngle + cameraAngle(observationTime)
```

Kalman估计的是仅消除相机旋转后的相对视线角和角速度。角色平移与目标平移的合成视线运动必须保留；没有角色世界速度和目标深度时，不能称为目标自身世界速度。控制时刻再计算：

```text
relativeError(controlTime)
  = predictedRotationStabilizedLosAngle(controlTime) - cameraAngle(controlTime)
```

这样可建模相机旋转和相对平移视线运动通过同一个时间模型分离，不再需要预测器内部判断“这次速度像不像自身视角伪迹”。角色自身平移仍会改变相对视线，应由`relativeLosRate`自然反映，不能按WASD方向直接扣除。

### 命令到画面的响应模型

首版使用最小模型即可：

- 固定总响应延迟`commandToFrameDelayMs`
- 可选一阶响应展开时间`commandResponseTauMs`
- 每个游戏、捕获链路和输入设备形成独立标定结果

标定时保持目标静止，记录发送角度脉冲/自然控制输出和观测目标反向角位移。使用互相关寻找两条序列峰值对应的时间lag，再用最小误差拟合比例与可选响应时间。SciPy官方相关函数明确支持通过互相关最大值获得信号相对延迟。

在线自适应应暂缓，因为移动目标会污染相机响应估计。先使用static标准视频或实机静止序列离线标定，并记录延迟P50/P95判断固定模型是否足够。

### 延迟观测的处理顺序

WPILib的延迟视觉融合采用已知时间戳将视觉测量应用到对应历史状态，再把后续控制/里程计输入重新推进到当前时刻。Xen可采用同一原则：

1. 在观测时间预测或回放到对应状态。
2. 用该时刻角度测量修正状态。
3. 重放观测之后已经发生的相机输入。
4. 将目标状态预测到当前控制时刻。

首版观测仍按顺序到达时不需要通用OOSM框架，只需保存覆盖最大端到端延迟的短历史环形缓冲。只有后续发现网络帧乱序，才增加真正的乱序测量处理。

## 前馈与反馈的最小实现

当前“目标速度 × 前瞻时间”生成框外预测点，再由位置控制器追逐该点。下一代改为直接组合角误差反馈与相对视线角速度前馈：

```text
feedbackAngle = errorAngle × (1 - exp(-dt / responseTime))
feedforwardAngle = feedforwardGain × relativeLosRate × dt
requestedAngle = feedbackAngle + feedforwardAngle + integralAngle
```

- 反馈项纠正准星与目标之间的角误差。
- 前馈项维持目标与角色平移合成后的持续相对视线速度，不需要不断把目标点推到框外。
- 积分项只处理可信持续偏差，并继续保留抗饱和与反向解卷绕。
- NIS、状态协方差或模型概率升高时，优先衰减前馈；反馈仍可把准星保持在框内。
- 输出统一由角度转换为counts，再执行二维总速度限制。

这比完整PIDF更适合现状。D项对检测噪声敏感，相对视线角速度已经由状态估计器提供，额外对误差求导会重复引入高频噪声。

## SOT与光流的正确接入位置

OpenCV资料表明，Lucas-Kanade光流依赖相邻帧亮度基本不变和邻域运动一致，金字塔可处理较大位移，但特征消失时可能错误匹配；官方示例建议周期性重建特征并使用反向检查。

因此若后续需要填补约238 FPS NDI与约100 FPS DML检测之间的空档，应采用：

- 只在最近一次可靠检测框内部选特征。
- 前后向光流一致性、有效特征比例和流向离散度共同形成置信度。
- 新检测到达时立即重置光流基准。
- 光流测量使用高于检测器的R，不能与真实检测等权。
- 框出界、特征不足、NIS异常或目标ID变化立即停止。

它属于观测补点，不负责决定提前量；在时间轴和角度Kalman通过前不应开发。

## 更新后的最小重构顺序

### 阶段0：只做时间轴与离线标定

1. 明确并记录`frameReceiveTime/inferencePublishTime/controlTime/commandQueueTime/deviceSendTime`。
2. 以static数据互相关估算`commandToFrameDelayMs`，检查不同链路P50/P95。
3. 区分已知本机观测年龄、上游视频年龄和用户额外提前量，禁止继续共用一个50 ms参数。
4. 使用Mouse Sensitivity数据校验每游戏`degreesPerCount`和双精度FOV。

### 阶段1：离线角度估计影子模式

1. 将raw pivot转换为yaw/pitch相对角。
2. 用延迟后的相机角历史形成稳定目标角观测。
3. 同时运行One Euro、Alpha-Beta和常速度Kalman。
4. 输出角度、角速度、协方差、innovation、NIS和未来控制时刻预测，不改变实际鼠标输出。

### 阶段2：离线前馈控制

1. 在回放器中加入“角反馈 + 角速度前馈”。
2. 与r30预测点直追使用完全相同的观测轨迹比较。
3. 单独搜索Q/R、反馈响应和前馈增益，禁止一次搜索全部参数。

### 阶段3：实机影子与小流量启用

1. 先只写CSV，不改变控制输出，确认实机NIS和延迟分布与离线一致。
2. 先在static和单向left/right启用新控制，再验证reverse和jump。
3. 新链路任何场景不达标时可立即切回r30，不在新旧链路之间混用门控。

### 阶段4：按证据增加复杂度

- 单Kalman机动响应不足：增加创新驱动Q切换或轻量IMM。
- 检测帧间空档仍是瓶颈：增加带置信度的光流/SOT。
- 获得可靠深度和武器参数：另建三维弹道层，不污染基础视线角追踪。

## 建议验收指标

除现有像素P95外，新链路必须增加：

- 角误差P50/P95/P99，单位mrad或度
- 角速度误差与符号翻转次数
- innovation与NIS分布
- 状态角速度标准差
- 前馈占总输出比例
- 前馈方向翻转次数
- command-to-frame延迟P50/P95
- 观测时间戳来源与完整端到端年龄
- 框内比例、目标丢失前边缘余量
- static中相机命令停止后残余相对视线角速度衰减时间

跨游戏验收不应要求像素速度相同，而应要求在不同FOV、分辨率、FPS和轨迹速度缩放下，角度误差、NIS和前馈稳定性不发生数量级变化。

## 本轮结论

继续检索后，最重要的新结论是：r30的跨场景门控问题不能只靠更换滤波器解决，必须先修正相机命令与观测画面的时间关系。推荐首版采用控制感知的线性角度Kalman，不串联One Euro，不使用UKF或粒子滤波；相机命令通过离线标定的command-to-frame响应模型进入稳定坐标，控制输出改为角误差反馈加角速度前馈。IMM和光流只在基础链路通过后按数据收益增加。

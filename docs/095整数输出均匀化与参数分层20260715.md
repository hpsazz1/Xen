# 整数输出均匀化与参数分层（2026-07-15）

## 背景

最新 DML+NDI 五类数据仍能看到右移和低速反转的周期性晃动。队列丢弃、发送延迟和预测中断均不是主因；右移 CSV 中连续控制量在相邻帧约为 0.7 与 5~7 counts，但最终整数命令随 10/20 ms 观测间隔交替，形成明显脉冲。

## 修复

legacy 链路在最终 `round` 前增加每轴分数 counts 余量。当前帧的小数部分跨帧保留，累计到整数才发送；目标丢失、目标切换和进入稳定停发时清零。该改动不增加配置项，不改变 P、积分、预测或设备速度上限，只把相同的长期位移均匀分布到相邻帧，适配 NDI 不规则帧间隔。

## 参数复核

当前真正影响 legacy 设备输出的用户参数只有三类：

1. `move_response_ms`：响应与稳定性的总开关；
2. `move_max_speed_cps`：设备计数速度上限；
3. `move_integral_time_ms`：移动目标固定滞后的补偿强度，设为 0 即关闭。

预测的 `prediction_lead_ms`、`prediction_velocity_tau_ms` 和 `prediction_strength` 是同一常速度预测模型的三个低层旋钮，现场不应独立扫描。保留旧配置键用于兼容，但推荐固定为经过验证的 50/50/1.0 组合，仅在专项回放中调整。`aim_shadow_*` 与 `trajectory_*` 当前属于 shadow/诊断链，`aim_pipeline_mode=legacy` 时不会进入设备队列，应从日常调参界面移到高级诊断区。

## 验证

代码测试需覆盖分数 counts 的累计发送、稳定停发清零和目标切换清零。完成 DML Release、CUDA Release 及两套 CTest 后，再用同一五类 NDI 场景复测 `FinalMx` 翻转频率、低速反转滞后和 jump P95；不能只凭静态 CSV 宣称现场问题已完全消失。

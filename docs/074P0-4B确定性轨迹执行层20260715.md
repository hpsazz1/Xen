# P0-4B确定性轨迹执行层

## 背景与目标

P0-4A已经把新控制器输出拆为双精度counts请求，但轨迹结构仍是占位。旧WindMouse、随机Perlin、固定端点Bezier和输出EMA会分别引入随机控制噪声、过期端点路径、多子步队列突发和额外相位滞后，不能直接恢复。P0-4B需要把设备工程约束实现为独立、确定性、可中止和可审计的在线执行层，同时保持r30正式输出不变。

本阶段新增`CommandTrajectoryShaper`和`OutputScheduler`。它们只生成shadow候选，继续设置`commandSuppressed=true`，不访问`MouseThread::moveQueue`、鼠标后端或P0-2实际成功counts历史。

## 请求与时间契约

P0-4A每个控制请求新增：

- `valid`与单调`sequence`；
- X/Y双精度请求counts；
- 产生该位移的真实控制周期`requestDurationSeconds`；
- 同一`steady_clock`域的`requestTime`。

整形器先使用`requestedCounts / requestDuration`得到目标二维速度。`off`模式不使用固定调度器，逐请求保持双精度位移完全相等；`trapezoid`模式由固定周期调度器把最新目标速度在线重规划为每tick位移。

## Pass-through基线

`off`是默认模式。`shapedCountsX/Y`逐帧严格等于P0-4A请求，证明接入接口本身没有改变控制语义。整数设备counts通过带符号余量量化：

```text
integer = round(shaped + previousRemainder)
newRemainder = shaped + previousRemainder − integer
```

因此连续四次`0.25 count`最终输出总和为`1 count`，负方向同样守恒；双精度请求和量化结果分别记录，不能用整数舍入误差评价控制器。

## 固定周期与latest-only

`OutputScheduler`维护固定输出格点和唯一latest request：

- 每次`service()`最多产生一个二维输出；
- 服务线程迟到时跳到不晚于当前时间的最近格点，记录跳过tick，不补发过期子步；
- 新请求只能从其到达后的首个格点生效，禁止回填到过去tick；
- 新观测覆盖旧目标速度，整形器从当前速度/加速度状态在线重规划，不保存端点路径。

当前shadow运行时随控制观测机会性服务该格点；独立类的固定周期语义由单元测试以30~1000 Hz调用验证。P0-6正式接管时必须由固定输出线程调用，并把真实设备队列改为latest-only；P0-4B不会在shadow阶段提前改变r30队列行为。

## 二维速度、加速度与jerk约束

`trapezoid`模式所有限制都作用于X/Y向量模长而非逐轴独立裁剪：

1. 目标速度限制为`maxVelocity`；
2. `(targetVelocity-currentVelocity)/dt`限制为`maxAcceleration`；
3. 相邻加速度差限制为`maxJerk×dt`；
4. 新速度再次应用二维速度安全上限；
5. 周期位移使用首尾速度平均值进行梯形积分。

为避免速度已经贴近上限且仍带正加速度时出现“速度球、加速度球和jerk球无交集”，实现还把单周期加速度保守限制为`min(maxAcceleration,maxJerk×dt)`。这样零加速度始终在下一周期jerk可达集合内；若候选速度仍越界，就沿候选加速度到零的线段二分最大可行比例。保存的加速度始终等于实际速度差分，保存的jerk始终等于实际加速度差分，不通过只投影速度来伪造满足约束的状态。

CSV记录目标速度、实际速度、加速度、jerk以及三个限制命中标记。目标丢失、ID切换、配置变化或设备失败调用紧急重置，单周期把速度、加速度、jerk和量化余量全部清零；紧急停止不伪装成普通受限减速。

## Ruckig评审与选择

Ruckig官方仓库显示Community版使用MIT许可证，支持本地CMake集成、velocity-control、固定控制周期以及速度/加速度/jerk约束；中间waypoint才会触发Community云API，本项目不会使用waypoint。官方仓库当前Latest为`v0.17.3`（2026-03-24），但当前主线说明使用C++20，C++17版本需另行联系；Xen当前构建固定`C++17`。官方任意信号Tracking接口又属于Pro版本。

因此本阶段不加入Ruckig源码、二进制或运行时依赖：当前简单约束器已经满足基础二维边界，尚无P0-5回放证明Ruckig能在相同限制下降低jerk/反转冲击且额外延迟不超过5%。配置读取到`ruckig`或未知模式时安全降级为`off`。只有P0-5发现简单约束器存在可测超调、延迟或同步缺陷时，才在独立候选提交中固定版本并重新完成许可证、C++标准、离线构建和双后端审查。

## 配置、UI与CSV

新增`trajectory_shaper_mode`、`trajectory_output_hz`、`trajectory_max_velocity_cps`、`trajectory_max_acceleration_cps2`和`trajectory_max_jerk_cps3`。UI位于“瞄准 → 移动参数 → 确定性轨迹影子参数”。

CSV新增三组字段：

- 请求与输出时间线：请求/输出序号、请求周期、计划格点、实际service时间、整形延迟、调度迟到和跳过tick；
- 运动状态：目标速度、实际速度、加速度、jerk和累计双精度位置；
- 执行结果：模式、是否产出、三个限制标记、紧急重置、双精度位移、整数counts与量化余量。

## 测试与验证

单元测试覆盖pass-through逐请求相等、正负整数余量长期守恒、二维速度/加速度/jerk边界、反转在线重规划、紧急归零、固定周期单输出、迟到跳tick、latest request覆盖和禁止请求时间回填。配置测试覆盖默认值、持久化、范围裁剪及未集成Ruckig安全降级；UI与CSV字段均有静态回归检查。

程序提交`8fec51edf8d9`完成最终干净构建：DML Release与CUDA Release均成功，两个后端CTest各8/8通过。`build/dml/Release/Xen.exe`与`build/cuda/Release/Xen.exe`均内嵌`8fec51edf8d9`且无`-dirty`；UI标题和CSV继续由同一`BuildIdentity`读取后端、提交和控制器r30，三处身份来源一致。构建仅出现既有OpenCV模板警告，无新增编译错误。

## 后续

P0-5需要在同帧回放中比较`off`与`trapezoid`的static、left/right、reverse和jump误差、反转冲击、整形延迟与跳tick。P0-6只有在回放通过后才允许固定输出线程接管，并必须证明真实`QueuedMoveCount P99≤1`和整形后实际成功counts进入P0-2相机响应模型。

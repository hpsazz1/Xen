# KMBOX 监听就绪等待

## 背景

冒烟测试中曾出现 `CommandSucceeded=1` 但首个 32 counts 脉冲没有画面位移，而此前连续扫描在设备已使用后正常。检查发现 KMBOX_NET 的 `isOpen()` 只表示 `kmNet_init` 完成，后台 `kmNet_monitor(10000)` 仍可能没有完成；在监听初始化结束前发送首个动作存在被设备忽略的窗口。

## 修复

输入接口新增 `isReadyForMotion()`，普通设备默认等同于 `isOpen()`。KMBOX_NET 只有在监听线程收到成功返回后才报告动作就绪，孤立采集器等待该状态后再开始基线和脉冲采集。监听失败或超时会安全退出，不发送脉冲。

## 验证

需要重新完成 DML/CUDA Release 构建和 CTest，并用固定场景运行一次 `counts=32` 冒烟测试。重点检查首轮是否出现与连续扫描一致的约 16 px 位移。

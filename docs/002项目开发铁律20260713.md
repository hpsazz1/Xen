# 项目开发铁律（2026-07-13）

## 目的

建立可执行、可检查、可沉淀的开发流程，避免局部补丁、文档缺失、一次性脚本、重复造轮子和无法复现的测试结果。

## 开发闭环

每项开发任务必须按以下顺序完成：

1. 全局审计：确认入口、数据流、线程、配置、UI、日志、构建和测试影响。
2. 顶层设计：明确核心目标、非目标、模块边界、参数单位和失败回退。
3. 最小实现：围绕核心功能形成最小可用闭环，非阻塞细节进入 `docs/todolist.md`。
4. 文档同步：代码、配置、流水线字段和使用方式发生变化时同步更新文档。
5. 自动验证：运行正式单元测试和构建脚本，禁止依赖一次性测试脚本。
6. 人工验证：需要设备或真实场景时，明确采集步骤和验收指标。
7. 版本身份核对：代码或构建相关改动需确认 UI 标题、流水线 CSV 和构建产物使用同一身份来源，并对应准备提交的代码。
8. Git 提交：检查改动范围后使用中文提交信息提交。
9. 云端推送：代码、构建、配置逻辑或运行资源改动需推送当前分支到已配置远端，并核对远端引用已经包含本次提交；纯非代码变更无需推送。

## Git 规范

- 每次代码变更必须提交；不得长期保留无法说明来源的大量未提交代码。
- commit 使用中文，说明实际结果，例如：`重构基础目标跟踪与移动控制`。
- 代码、构建脚本、配置逻辑、项目文件和运行资源变更在本地提交后必须推送当前分支到云端远端，禁止只提交本地不推送。
- 仅修改 `AGENTS.md`、`docs/*.md`、说明文字或待办清单的纯非代码变更允许只做中文本地提交，无需推送；用户明确要求推送时除外。
- 推送失败必须继续排查或明确报告阻塞；只有远端引用与本地提交一致时，Git 闭环才算完成。
- 不把构建产物、临时 CSV、个人配置和一次性诊断文件提交到仓库。
- 提交前至少执行 `git diff --check`，并记录相关测试结果。

## 文档规范

- 所有新项目文档放在根目录 `docs/`。
- 新文件命名格式：`序号中文文件名日期.md`，文件名不得含空格。
- 示例：`003流水线计数异常排查20260713.md`。
- 历史文档保持现状，后续修改时再评估迁移，避免无价值的大规模重命名。
- `docs/todolist.md` 是固定例外，只记录不阻塞当前核心功能的待办。

踩坑记录必须包含：

- 背景与真实场景；
- 现象和可复现步骤；
- 根本原因；
- 排查过程与被排除的假设；
- 最终解决方案；
- 测试和验证结果；
- 防止复发的建议。

## 设计原则

- 保持全局视野，优先解决职责边界和数据流问题，不连续追加局部特判。
- 做小而美的工具，不构建与核心目标无关的大而全平台。
- 参数必须有单一、稳定、可解释的含义，优先采用时间、像素、counts/s 等明确单位。
- 优先使用成熟行业方案；自研前必须记录现有方案不适用的理由、收益和维护成本。
- 所有方案必须可在真实运行环境验证，不以抽象口号代替数据。

## 测试和脚本规范

- 核心算法必须覆盖正常、边界、异常输入、重置和跨帧率行为。
- 共享 C++ 代码变更原则上同时验证 DML 与 CUDA Release。
- 构建和测试入口必须使用仓库内正式脚本；有复用价值的诊断命令整理进脚本或指南。
- 每次正式构建必须让 UI 标题、流水线 CSV 和可执行文件共享同一版本身份，至少显示后端、Git 提交和控制器修订号。
- 提交后必须重新构建最终产物，确认版本身份对应当前干净提交；不得将 `-dirty`、旧提交或历史 `x64` 目录程序作为交付结果。
- 禁止为了单次调试反复创建、删除临时脚本；确需临时实验时，结束前决定沉淀或清理，并在踩坑文档保留方法和结论。

## 提交与云端同步检查

代码、构建、配置逻辑或运行资源任务结束前至少完成：

1. `git diff --check` 和相关测试。
2. DML Release 构建；共享代码同时完成 CUDA Release 构建。
3. 核对 UI/CSV/构建产物版本身份。
4. 使用中文提交信息完成本地 commit。
5. 提交后重新构建最终产物，使其写入当前干净提交号。
6. `git push` 推送当前分支到云端远端。
7. 获取远端分支引用，确认其提交号与本地 `HEAD` 一致。

纯非代码变更适用精简闭环：

1. 确认只涉及 `AGENTS.md`、`docs/*.md`、说明文字或待办清单，不包含源码、脚本、CMake/项目文件、配置默认值、资源、清单或运行行为。
2. 检查文档内容、引用和待办是否与当前实现一致。
3. 执行 `git diff --check`。
4. 使用中文提交信息完成本地 commit。
5. 无需运行单元测试、DML/CUDA构建、提交后版本重建或云端推送；用户明确要求时再执行。

## 工具使用

在适用且环境可用时优先使用 RTK、MCP 和项目 Skills，以减少重复检索和上下文消耗。工具只是效率手段，结果必须由代码、日志、测试或权威文档验证；工具不可用时应明确采用的替代方案。

## 代码注释规范

核心代码、变量、常量和业务流程使用中文注释说明：

- 为什么存在；
- 单位和坐标系；
- 边界与不变量；
- 失败回退；
- 与上下游模块的契约。

禁止只把代码语句翻译成中文，也禁止注释与实现长期不一致。

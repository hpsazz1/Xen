# 正式版本收口核对

## 核对范围

本次按计划重评结论执行正式版本的轻量收口核对，目标是确认当前发布基线是否需要重建，以及关闭待办中的稳定版收口和真实使用问题回收项。本次不修改源码、配置、资源或运行行为。

## 核对结果

### 正式模式与速度

- `Xen/config/config.h`和`Xen/config/config.cpp`默认`aim_pipeline_mode=legacy`，非法模式也回退到`legacy`。
- legacy正式路径读取`move_max_speed_cps`，源码默认值为3200 counts/s；shadow候选读取独立的`aim_shadow_max_speed_cps`，默认值为1440 counts/s。P0-5文档中的“正式1440”描述的是新控制候选相对1800恢复建议的基线，不得覆盖legacy正式配置。P0-5结论没有把1800写入任一路径配置。
- `active`在P0阶段只保留请求状态并安全降级为shadow；`ActiveAutoEnable=0`已由设备协议决策记录确认。

## 发布结论

源码和程序行为的正式模式继续使用：

- 设备输出路径：`legacy`；
- 影子诊断：可选，只读且不接管设备；
- `active`：关闭，不作为正式发布模式；
- legacy速度配置：保持现有`move_max_speed_cps`，源码默认3200 counts/s；
- shadow候选速度：1440 counts/s，仅用于零输出诊断；
- P0-5：保持`HOLD_SHADOW`，不追加算法或参数扫描。

但发布物核对尚未完成：`build/dml/Release/config.ini`实际为7月17日遗留的`aim_pipeline_mode=shadow`测试快照，且该文件和整个`build/`目录均被`.gitignore`忽略；`build/cuda/Release`没有对应配置。它们不能作为正式发布配置交付。下一步仅需生成或确认一份`aim_pipeline_mode=legacy`的正式配置，并再次核对UI、CSV和EXE身份；不需要重开P0-5、不需要重新采集实机数据。当前没有已知真实使用问题；P1改为长期触发规则。

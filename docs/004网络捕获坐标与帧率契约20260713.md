# 网络捕获坐标与帧率契约（2026-07-13）

## 背景与原因

双机 UDP/NDI 场景中，主机游戏分辨率、网络传输分辨率、检测分辨率和辅机显示器分辨率是四个独立概念。旧实现把任意网络帧拉伸为正方形检测尺寸，并把检测尺寸登记为完整源尺寸，导致宽高比失真和 FOV 换算错误；网络队列按最旧帧消费还会增加控制延迟。

## 设计

- 辅机显示器尺寸只负责界面显示，永不参与瞄准换算。
- NDI/UDP 解码后的完整网络帧尺寸表示当前视频所覆盖的完整 FOV。
- 接收端从完整帧中心按 1:1 像素裁出检测区域；仅当网络帧小于检测尺寸时才做安全放大。
- 接收队列每次丢弃旧帧并只返回最新帧。
- 控制器优先使用相邻有效观测时间戳计算逐帧 dt；捕获窗一秒平均 FPS 仅作首帧回退与界面统计。
- 裸 MJPEG UDP 不携带原始 ROI 元数据，因此当前兼容模式要求发送完整画面。发送端预裁 ROI 必须在未来版本化协议中附带完整源宽高、ROI、帧号和时间戳。

## 实现与验证

NDI 和 UDP 均保存解码帧报告的完整宽高，将中心裁剪后的图像送入检测器，并在取帧时发布源尺寸。流水线 CSV 已记录 `SourceWidth/SourceHeight`，可验证主机视频尺寸不受辅机显示器影响。

提交前执行 `git diff --check`、基础算法单元测试、DML Release 和 CUDA Release 构建。

## 后续建议

UDP 下一阶段应增加带版本号的分片帧头，包含源尺寸、ROI、帧序号、发送端帧间隔和校验；NDI 若由发送端预裁 ROI，也应通过 NDI metadata 传递同样字段。在协议落地前，不允许把预裁 ROI 冒充完整视频帧。

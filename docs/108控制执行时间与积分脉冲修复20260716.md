# 控制执行时间与积分脉冲修复

## 背景与复测结论

r46以DML、NDI、正式补偿12/0 ms完成left/right各三轮复测。left/right的X真实稳态P95分别为12.33/12.83 px，X/Y最终输出换向均为0，预测生效率为99.1%/98.5%，成熟段预测中断均为0。说明r45出现的预测复位已经解决，本轮不再修改预测门控或自运动模型。

残余问题是同方向幅值震荡：left/right最终输出同方向步进P95为7.33/8.00 counts，局部脉冲频率为35.4/34.0 Hz。连续`RequestedCountsX`步进P95为7.37~7.64 counts，`IntegralCountsX`步进P95为7.17~7.35 counts，证明大幅交替在整数化之前已经存在，整数余数不是主因。典型输出为积分`-6.x -> -0.8 -> -6.x`，最终命令`-7 -> -1 -> -7`。

## 根因

NDI的`BackendReceiveNs`观测间隔呈明显成批到达：left/right P05为0.45/0.63 ms，P50为13.88/14.05 ms，P95为16.34/16.26 ms。相邻`ControlTimeNs`控制执行间隔则为P05 8.61/8.57 ms、P50 9.34/9.31 ms、P95 14.46/14.50 ms。

状态估计必须使用观测时间，才能正确解释目标在哪个时刻被看见；但基础PI控制器输出的是本次设备命令应覆盖的counts，必须使用相邻控制执行时刻。旧实现把相邻`observationTime`作为PI的`dt`，因此0.5/15 ms的网络批次间隔直接把连续速度调制为约`-0.8/-7` counts。

## r47设计与实现

新增`ControlIntervalTracker`，以相邻`controlTime`计算输出周期，首帧使用捕获FPS回退，并将异常周期限制在2~50 ms。跟踪状态重置时同步清空控制周期历史。观测时间仍只供滤波、预测和观测年龄使用，两个时间域不再混用。

流水线CSV新增`ControllerUpdateIntervalMs`，记录基础控制器实际采用的钳制后周期。移动分析器新增请求输出步进、积分输出步进、控制周期P05/P95和观测周期P05/P95；新CSV直接读取控制周期字段，旧CSV可由`ControlTimeNs`回算，缺少可选字段时保持兼容并报告0。

## 验证与下一步

单元测试覆盖首帧回退、12 ms相邻控制周期、过短周期钳制、reset语义、CSV字段和分析器汇总。代码提交前后执行DML/CUDA Release构建与两套完整CTest，并核对UI、CSV、构建对象和可执行文件统一为r47及同一干净提交。

首次正式构建在受限环境中暴露NuGet客户端启动时强制读取用户漫游配置的问题。DML正式脚本新增显式`-SkipNuGetRestore`离线入口，跳过前按`Xen/packages.config`逐项核对固定版本包目录；默认联网恢复行为不变。依赖解析清单同时从共享的`build/dependency-resolution.json`拆分到各后端构建目录，避免陈旧ACL和双后端写入争用。

全新输出目录的发布布局测试还发现NDI运行库过去依赖旧目录残留，CMake仅链接SDK而未复制DLL。现已在启用NDI时从SDK `Bin/x64`强制复制`Processing.NDI.Lib.x64.dll`，缺失时构建直接失败，保证干净Release具备完整运行闭包。

下一轮继续使用DML+NDI、12/0 ms配置完成left/right各三轮。验收要求：`ControllerUpdateIntervalMs`与`ControlTimeNs`一致且不再跟随NDI批次；积分、请求和最终输出步进P95显著低于r46约7.2/7.5/7~8 counts，脉冲频率低于35.4/34.0 Hz；预测中断、X/Y输出换向保持0，X真实稳态P95不高于12.33/12.83 px。

# P0-5恢复速度建议影子诊断

## 背景

184号离线反事实表明，1800 counts/s仅在目标丢失后的恢复窗口内使用时，可以在179号端点守卫的四端点共同485项上保持零损失，并把共同通过域扩大到689项。但该结果不代表设备能力、运行时策略或发布授权。本阶段只把同一反事实做成shadow旁路诊断，用真实CSV回答“若上限为1800，本帧会怎样裁剪、静态请求预算可能减少多少”，正式控制仍固定为1440 counts/s。

## 调用链与边界

`mouse_thread_loop`把启动、配置重载、跟踪器切换、目标ID切换和真实目标丢失区分开。只有此前已经存在有效观测且随后真实丢失，才通过`resetTracking(true)`武装下一reset代次；初始空检测、连续跟踪、配置变化和ID切换均不能启用建议。

`AimPipelineRuntime`先运行原`LosAimController`并写入正式`control.requestedCounts`与1440帧限额，之后才派生`RecoverySpeedAdviceDiagnostics`。建议结构只读取无限制请求、正式帧限额和同帧`dt`，按二维模长计算固定1800 counts/s的建议裁剪；它不覆盖正式请求，不修改`TrajectoryRequest`，不进入轨迹整形，也不接触设备队列。active请求仍降级为shadow，两个命令抑制位保持为真。

本次CSV契约和恢复重置语义使控制器修订由r64升为r65。严格机器Profile缓存会按既有失效键拒绝把r64证据当作r65精确命中，并安全降级；不得手工篡改旧Profile的修订号。该失效只影响证据身份，不会把1800建议接入正式输出。

## 状态机

1. 真实目标丢失后，下一代首个有效观测进入建议合格代次。
2. 正式1440控制仍限速时，建议处于active并计算1800帧限额与二维裁剪。
3. 正式请求首次不再限速时，退出帧本身保持正式1440行为并记录`Exited=1`；同代次后续即使再次出现大请求也不得重启建议。
4. 再次目标丢失会结束当前代次，并在下一次恢复建立新建议代次。
5. `BaselineStaticBudgetFrames`与`AdvisoryStaticBudgetFrames`分别为`ceil(同帧无限制请求模长/帧限额)`。它们只表示静态请求预算，不是闭环真实退出帧预测。

## CSV与分析工具

逐帧CSV新增独立的`RecoverySpeedAdvice*`字段块，包含合格、启用、退出、建议仍限速、1440/1800身份、建议帧限额、建议二维请求和静态预算。原`AimPipelineRequestedCounts*`、`AimPipelineFrameCountLimit`、轨迹字段和命令生命周期字段含义不变。

正式分析命令：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File tools/analyze_recovery_speed_advice.ps1 `

  -ExpectedBuildRevision <r65构建提交短哈希> `
```

工具拒绝混合构建身份、缺列、非1440正式身份、非1800建议身份、非合格帧启用、正式请求越过1440帧限额或建议请求越过自身诊断限额。无违规时结论固定为`DIAGNOSTIC_ONLY_HOLD_SHADOW`，不会产生晋级结论。

## 测试与验证

核心C++测试覆盖：初始代次不启用；真实丢失后启用；正式与建议二维裁剪分别服从1440/1800；轨迹请求逐值等于正式请求；退出后同代次锁定；非丢失重置不伪装恢复；命令抑制不解除。PowerShell合成测试覆盖双方向建议帧、退出帧、静态预算和安全结论。构建身份与最终测试结果在提交后的清洁双后端构建完成后补录。

## 结论与下一步

本阶段只建立可审计建议，不提高设备速度，不新增配置或UI，不开放active，P0-5继续`HOLD_SHADOW`。下一步需要使用本次r65 DML清洁构建人工采集新的left/right完整丢失与恢复CSV；旧r64文件不含新增字段，不能验证运行时建议状态。

采集长度不再固定为20帧。每个方向至少覆盖当前实测最长的正式1440退出路径，并持续记录到`RecoverySpeedAdviceExited=1`后至少5个有效观测；若现场实际退出晚于既有最长文件，则以新的实际退出帧数延长采集。只有新CSV证明建议仅在恢复代次启用、正式请求始终受1440约束、每个完整事件均自然退出且分析器零违规后，才允许继续评估建议静态预算与真实退出帧的关系；仍不得向设备发送1800命令。

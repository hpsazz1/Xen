# r61有限响应首轮实测

## 背景与数据

r61把DML shadow应用相机模型从固定阶跃改为中心20 ms、宽度20 ms的线性响应，目标是在不调整`maneuver_gated_ca + 8000/12/120`的前提下消除九点static运行态机动误启用。本轮分析`C:\Users\16143\Desktop\Xen\DML\ndi\static.csv`和同目录实际`config.ini`；单个CSV内包含九个按键段，必须按`AimPipelineOutputPaused: 1→0`边界拆分，不能按文件数解释轮次。

配置与逐帧身份一致：`DML / cc8093b6f73c / r61`、`aim_pipeline_mode=shadow`、`CommandToFrameDelayMs/CommandResponseMs=20/20`、门控CA参数为8000/12/120，被动Profile标定关闭。CSV共2130帧，其中运行态1266帧、暂停态864帧，准确切分出九个运行段。

## 自动门禁

正式Shadow分析器使用`-RequirePausedObservations -RequireManeuverCandidate -RequireFiniteViewResponse`验收。身份、模式、时间顺序、估计有限性、暂停命令、暂停连续观测、模型选择计数和20/20响应契约均为0违例；暂停态有211个机动启用样本，只反映换位期间持续影子观测，不计入运行态static失败。

运行态仍有37个机动启用样本，因此整体为`FAIL`。37帧集中在第1、3、7段，分别为15、10、12帧；其余六段为0。按冻结九点顺序对应左上、左下、右上，首次启用发生在恢复后75、59、69 ms，最后启用发生在219、160、183 ms。共有16帧已经进入legacy收敛态，不能作为单纯启动热身忽略。

运行态最大模型角度差为0.574°、最大模型速率差为22.26°/s；相对固定20 ms阶跃历史数据的0.950°和41.40°/s，有限响应降低了单次模型分歧幅度，但误启用从2/9段28帧变为3/9段37帧，且收敛态侵入从12帧变为16帧，没有通过安全门槛。

## 逐轴与时序归因

三个失败段的估计速率峰值几乎全部位于X轴：X为39.85、25.63、34.97°/s，Y全局峰值仅5.55°/s。触发前后可见检测中心保持或阶跃更新时，线性相机模型仍按连续时间累积命令，恢复后约60~75 ms形成稳定LOS错相；这不是暂停态旧模型直接泄漏，恢复首帧也没有继承机动启用。

使用CSV中的真实设备发送时刻、Backend接收时刻和counts做响应核反事实后，20/20相对20/0确实降低全段静止世界角速度RMS，但20/10与20/20差异极小。只对恢复前250 ms按原始相邻帧角速度评分又会受到检测中心离散跳变主导，给出与实机固定延迟A/B冲突的最优点，因此不能据此直接选择10 ms宽度或继续实机细扫。

本轮证明有限线性响应方向上降低了模型差异，却不足以消除X轴恢复瞬态。现有一份r61数据还不能区分固定点位复现与NDI发布相位波动；历史r60原始CSV已不在上传目录，无法在同一时间线上复放新旧核。

## 结论与下一步

1. r61继续`HOLD_SHADOW`，不得采集jump/reverse，不得开放active或CUDA候选。
2. 保留本轮`static.csv`和`config.ini`，再采集两份完全相同身份、配置和九点顺序的独立static文件，命名为`static_r61_repeat2.csv`和`static_r61_repeat3.csv`；每份测试前独立启动程序并重置一次流水线，九点之间只松开按键换位，不重置流水线。
3. 两份复测继续使用20/20和8000/12/120，不调整CA门槛、保持时间、legacy参数或Profile标定，不做10/15 ms宽度实机扫描。
4. 汇总三份文件后检查失败点位、首次启用时刻、X/Y峰值和收敛态侵入。若运行态仍不能稳定为0，则正式否决当前线性响应核，先增加可复用的逐轴/观测帧离线复放，再由同一原始时间线选择X轴或按观测帧对齐候选；禁止直接凭单轮RMS引入新参数。

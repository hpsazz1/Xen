# 瞄准释放晚到命令与末尾轻微复位修复

## 背景与复测结论

用户使用有效DML `f7c3b0d737fe/r48`重新完成horizontal_left/right各三轮，反馈每轮末尾仍有轻微复位感。六轮CSV身份一致，排除了上一轮混合旧对象问题。

正式分析显示left三轮预测中断、预测方向翻转和X/Y输出换向均为0；right仅第一轮启动后的约1秒内反复命中自运动门控并重建一次，之后持续有效。六轮最终输出同方向步进P95均为3 counts，left/right脉冲频率分别为1.88/2.07 Hz，真实稳态误差P95三轮均值分别为12.83/12.67 px。每轮最后150 ms内预测提前量仍稳定在约12~14 px，全部门控为0，目标和预测均持续有效；没有运行期预测复位证据。

CSV只在`moveMousePivot()`处理有效瞄准观测时提交，用户松开瞄准后立即停止记录。因此“每轮末尾”可感知事件发生在最后一条CSV之后，不能归因于预测状态机。结合队列生命周期，剩余现象定位为退出链路的晚到设备命令。

## 根因与调用链

正常控制帧由`moveMousePivot()`生成约1~3 counts命令，`queueMove()`将其交给异步Worker。Worker在持有`queueMtx`时弹出命令，随后释放队列锁，再进入`sendMovementToDriver()`调用硬件。

瞄准状态变化、目标失效和控制状态重置都会调用`resetTracking()`，后者原来只通过`clearQueuedMoves()`清除仍在`moveQueue`中的项目。如果Worker已经弹出最后一条命令但尚未调用设备，该命令不再位于队列中，旧清理逻辑看不到它；松开瞄准后仍可能晚到发送一次。由于r48稳态单帧输出正好约1~3 counts，这与用户感知的轻微末尾拖尾量级一致。

## r49设计与实现

新增线程安全`CommandCancellationEpoch`：命令入队时捕获当前取消代次；每次清理先递增代次，再清空队列。Worker在弹出后、取得设备锁后分别核对代次，旧代次命令不进入驱动并沿现有`CommandDroppedBeforeSend`诊断记录为发送前丢弃。

清理队列后再短暂取得设备锁作为完成屏障：若旧命令已经进入不可撤销的设备调用，清理等待该调用完成；清理返回后，旧代次命令不可能再晚到设备。锁顺序保持`queueMtx`先释放、再取得`inputDevicesMutex`，Worker发送路径不同时持有两把锁，避免引入锁反转。

本次不修改预测器门控、提前量、PI参数、控制周期或设备速度，只修复异步命令生命周期。控制行为身份升级为r49。单元测试覆盖取消前令牌有效、取消后旧令牌失效且新代次有效；现有流水线身份测试同步要求r49。

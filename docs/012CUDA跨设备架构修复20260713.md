# CUDA 跨设备架构修复

## 背景

CUDA Release 在构建机可以运行，复制到其他设备后，OpenCV `cudev` 的图像变换内核报错：`no kernel image is available for execution on the device`。实际使用还要求同一套源码分别生成 CUDA、DML Release，并对 UDP、NDI 两种网络采集方式进行组合测试。

## 原因与排查过程

发布目录中的 `opencv_world4130.dll` 与 OpenCV 安装目录 DLL 的 SHA256 完全一致。`opencv_version --verbose` 显示该 DLL 只有 `NVIDIA GPU arch: 120`，`build/ocv-cuda/CMakeCache.txt` 同样记录 `CUDA_ARCH_BIN=120`。

主程序 `build/cuda/CMakeCache.txt` 则记录 `CMAKE_CUDA_ARCHITECTURES=75`。这说明同一发布包中的 OpenCV CUDA 内核与 `cuda_preprocess.cu` 分别只面向构建时选中的单一架构，均不具备跨设备覆盖能力。

进一步检查发现：

1. `-CudaArchBin all` 原来只传给 OpenCV，没有传给主程序的 `CMAKE_CUDA_ARCHITECTURES`。
2. 构建脚本仅通过头文件、LIB、DLL 是否存在判断 OpenCV 可复用，不检查已安装 DLL 的 CUDA 架构。
3. 因此即使命令行指定 `all`，已有单架构 OpenCV 仍可能被直接复用并复制进发布目录。
4. 双后端验证时发现，部分自动化宿主同时向进程注入 `PATH` 与 `Path`。`VsDevCmd.bat` 返回的开发环境路径会被旧值覆盖，导致构建脚本误报缺少 `cl.exe`、`link.exe`、`rc.exe`、`mt.exe`。
5. CUDA 全量重编还发现 `keyboard_listener.cpp` 使用 `std::cout`、`std::cerr` 时依赖其他头文件间接包含 `<iostream>`。增量 DML 对象掩盖了问题，干净构建会稳定失败，因此改为直接包含标准头文件。

## 设计与实现

新增唯一的 CUDA 架构解析流程：

- OpenCV 使用带小数点格式：`7.5;8.0;...;12.1`。
- 主程序 CMake 使用整数格式：`75;80;...;121`。
- 两种格式从同一个架构集合生成，避免配置漂移。
- CUDA Release 默认选择 `all`，显式传入单架构或架构列表仍受支持。

OpenCV 安装根目录新增 `opencv-cuda-build.json`，记录 OpenCV 版本、CUDA Toolkit、`CUDA_ARCH_BIN`、主程序 CMake 架构及构建类型。主构建在复用 OpenCV 前必须验证该清单：

- 清单缺失、损坏或架构不一致：自动启用清洁重建。
- 同时指定 `-SkipOpenCvBuild`：立即报错，不生成错误发布包。
- 架构完全一致：安全复用，避免重复编译大型 OpenCV 依赖。

Visual Studio 环境导入会收集大小写不同的 PATH 候选，并优先选择包含 `VC\Tools\MSVC` 的开发环境路径，保证交互终端和自动化宿主采用相同工具链。

CUDA OpenCV 明确关闭 `WITH_FFMPEG` 与 `WITH_IPP`。UDP、NDI 都由项目采集器直接提供图像帧，不依赖 OpenCV FFmpeg；CUDA 推理也不依赖 CPU IPP。关闭这两项可以移除 OpenCV 配置阶段对可选预编译包的网络下载，避免网络波动阻塞可重复发布构建。

## 单元测试

`tests/build_script_tests.ps1` 覆盖：

- `all` 的 OpenCV 与 CMake 双格式展开。
- 显式架构列表的规范化和去重。
- 非法架构格式拒绝。
- 旧 OpenCV 无清单时禁止复用。
- 单架构清单不能满足多架构请求。
- 相同架构集合允许复用。
- 重复 `PATH`/`Path` 存在时正确选择 Visual Studio 开发环境路径。

该测试注册到 CTest，DML 与 CUDA 构建树都能执行同一套构建规则回归检查。

## UDP、NDI 双后端测试边界

UDP、NDI 属于采集输入，DML、CUDA 属于推理后端。四种组合共用捕获、分辨率/FOV、帧率统计与移动控制逻辑，但分别使用两个 Release 可执行文件：

| 推理后端 | UDP | NDI | 可执行文件 |
|---|---|---|---|
| DML | 需要实机验证 | 需要实机验证 | `build/dml/Release/Xen.exe` |
| CUDA | 需要实机验证 | 需要实机验证 | `build/cuda/Release/Xen.exe` |

本机构建验证只能确认程序、运行库、UDP/NDI 代码路径和测试通过。网络发送端、NDI 源发现、实际帧率与跨设备显卡执行必须在目标双机环境完成实测。

## 后续建议

每次 CUDA 发布前保留 `all` 构建及架构清单，不要用本机自动探测的单架构 DLL 覆盖发布依赖。目标设备若早于 Turing（例如 Pascal `6.1`），CUDA 13.2 已不在本项目发布集合内，应改用 DML 或单独维护兼容旧 Toolkit 的构建链。

## 验证结果

2026-07-13 在 CUDA 13.2、OpenCV 4.13.0、TensorRT 10.16 环境完成验证：

- 构建脚本单元测试通过。
- DML Release 构建通过，输出 `build/dml/Release/Xen.exe`。
- CUDA Release 构建通过，输出 `build/cuda/Release/Xen.exe`。
- DML CTest 3/3 通过。
- CUDA CTest 3/3 通过。
- OpenCV Build Information 显示 `NVIDIA GPU arch: 75 80 86 87 88 89 90 100 103 110 120 121`。
- CUDA 发布目录的 `opencv_world4130.dll` 与安装目录多架构 DLL 大小一致，均为 627,334,656 字节。
- DML、CUDA 发布目录均包含 `Processing.NDI.Lib.x64.dll`，UDP 路径由程序内置 Winsock 采集器提供，无额外运行库。

本机没有代替目标双机环境伪造 UDP 包、NDI 发送端、目标显卡型号和实际网络抖动。四种运行组合仍需按前述矩阵进行实机测试；本次验证保证两套待测 Release 已正确生成，且 CUDA 发布物不再局限于构建机单一架构。

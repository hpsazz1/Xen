# 短暂松开同目标预测快速恢复

## 背景与新数据

用户使用DML `f3bbdd5ce6c6/r49`在持续单向目标中加入“松开瞄准、短暂停顿、重新按住”操作，观察到重新追踪后不能第一时间产生预测，准星位于目标身后。

CSV按大于100 ms的控制记录空档识别短暂松开边界。left/right的主要短暂停顿为124~253 ms；重新按住时X误差通常已达到32~63 px。现有预测提前量恢复需要22~79个观测帧、约237~841 ms，典型right Frame 4184在203 ms空档后重按，直至Frame 4214、293 ms后才首次产生3.1 px提前；Frame 4301和4833之后分别延迟806/430 ms。用户描述与数据一致。

重按后的方向锁通常约40~60 ms建立，但预测器随即把追赶相机造成的屏幕运动识别为自身运动伪迹，进入多轮保持和运动证据重建；成熟提前量在此期间一直为0。问题不是设备速度或r49命令取消，而是瞄准状态变化无条件调用完整`resetTracking()`，主动丢弃了同一跟踪目标已经成熟的速度与方向。

## 设计约束

短暂松开必须立即停止设备输出，不能继续排队或发送移动；同时，只有可证明仍是同一跟踪ID、暂停足够短且端点运动仍同向时，才能复用预测状态。目标丢失、ID切换、长暂停、停止和反向必须保持原有冷启动安全语义。

350 ms窗口覆盖本轮124~253 ms操作，并明确排除870~1581 ms的长间隔。该窗口不是无条件滑行：未瞄准期间基础跟踪器仍持续更新身份；ID变化或目标丢失会立即完整重置。恢复首帧还要求跨暂停端点速度达到激活下限的50%且与成熟方向夹角余弦不低于0.50。

## r50实现

瞄准从按住变为松开时改调用`MouseThread::suspendAimingOutput()`：沿用r49取消代次清空并屏障设备命令，重置基础控制器、影子管线、控制周期和整数余量，同时保留同一目标的基础滤波与`TargetPredictor`成熟方向。为避免用松开前历史误判自身运动，单独清空自运动三帧历史和上一帧伪迹确认状态。

重新按住后的首个真实观测根据暂停时长选择最大观测间隔：不超过350 ms时允许滤波器使用真实跨暂停时间，预测器保留成熟速度；窗口暂时只有松开前末帧与恢复首帧时，只在端点速度仍同向的条件下容忍最多两个稀疏帧，随后正常四帧回归窗口接管。超过350 ms则在消费首帧前清空滤波与预测，行为等同原冷启动。

目标切换和目标丢失仍调用`resetTracking()`并清除快速恢复标记。预测参数、提前量、自运动几何门控、PI、速度预算和r49退出命令取消均未修改。控制行为身份升级为r50。

## 测试与复测要求

单元测试构造已成熟的500 px/s水平运动：跨200 ms同向空档并显式允许350 ms快速恢复时，恢复首帧必须保持方向锁和正向提前；跨808 ms长空档时必须清除方向与提前。流水线身份测试同步要求r50。

程序提交`e05b8b6d7f49`后，正式DML/CUDA包装脚本分别清理并重编译86/92个目标，两套完整CTest均为9/9通过，构建脚本单测通过。DML/CUDA构建定义分别记录后端`DML`/`CUDA`、统一程序修订`e05b8b6d7f49`和构建时间`20260716T102427Z`/`20260716T102455Z`；两个Release可执行文件均包含该修订且不包含`e05b8b6d7f49-dirty`。流水线测试确认`ControllerRevision=50`，UI、CSV与算法对象身份一致。

`git diff --check`通过。构建只保留CUDA SDK头文件既有C4819编码警告和历史`x64`旧可执行文件提醒，正式产物为`build/dml/Release/Xen.exe`与`build/cuda/Release/Xen.exe`。

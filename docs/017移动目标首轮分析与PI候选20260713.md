# 移动目标首轮分析与PI候选

## 背景

静止目标基线归档后，使用CUDA环境、80 ms响应、1440 counts/s上限，分别采集NDI和UDP的水平向左、水平向右及左右反转数据。用户随后将UI响应时间实测压到最小20 ms，仍观察到准星无法长期追到持续匀速目标中心。

本轮目标是区分设备限速、网络/推理延迟、发送队列、比例响应和移动目标结构性稳态误差，避免继续盲目降低响应时间或提高速度上限。

## 数据完整性

六份CSV均包含新版 `ObservedVelocityX/ObservedVelocityY` 字段、2560×1440完整FOV和1440 counts/s配置。单向文件各包含3轮有效采集。

UDP反转文件恰好1000行，命中原追踪环形缓冲上限，开头一部分数据已被覆盖；只识别到4次完整误差侧翻转。NDI反转文件保留813行和5次翻转，但该轮源到达约137 FPS、推理约85 FPS，明显低于NDI单向测试的约232～239 FPS源到达和106～169 FPS推理。因此两条反转数据可用于验证分析口径，不能直接把恢复时间差归因到NDI或UDP协议。

误差符号与移动方向一致：向左为负、向右为正，说明这是持续落后目标的跟随滞后，不是随机检测偏移。所有单向轮次的 `SpeedLimitedPct` 均为0，移动队列最大深度均为1；因此1440 counts/s设备上限和发送队列都不是当前瓶颈。

## 输出需求与闭环滞后

分析器通过 `FinalMx/My` 和真实采样时间计算实际输出counts/s，并从 `RequestedCounts / RequestedPixel` 中位数恢复当前counts/px。匀速单向场景使用以下关系估算闭环滞后：

```text
近似闭环滞后 = 平均绝对误差 × counts/px ÷ 平均绝对输出counts/s
```

| 链路 | 单向平均误差 | 平均输出需求 | 近似闭环滞后 |
|---|---:|---:|---:|
| CUDA + NDI | 53.1 px | 837.2 counts/s | 85.2 ms |
| CUDA + UDP | 47.8 px | 766.6 counts/s | 84.1 ms |

十二轮的实际目标跟随需求约576～874 counts/s，明显低于1440上限；折算滞后集中在80 ms响应附近并叠加少量观测延迟。两种传输归一化后几乎一致，原始P95差异主要来自每轮目标速度不完全相同，不能为NDI和UDP拆分控制参数。

## 根因

当前 `BasicAimController` 是比例型一阶控制。对静止阶跃输入，它能指数收敛；对持续匀速的斜坡输入，稳态误差满足：

```text
稳态跟随误差 ≈ 目标速度 × 响应时间
```

因此把响应时间从80 ms降到20 ms只能按比例缩小滞后，数学上不能让匀速目标长期零误差。用户观察到20 ms仍无法“追上”不是UI下限缺陷，而是纯比例控制器的结构性边界。继续提高 `move_max_speed_cps` 同样无效，因为本轮从未触发限速。

## 技术选型

不恢复旧预测/速度前馈链路。旧实现使用相对画面速度，包含自瞄自身转动造成的目标位移；在没有可靠相机运动补偿速度估计前直接前馈会放大检测突刺和反转误判。

采用成熟、最小的PI方案：在现有比例控制上增加误差积分，使含位置积分特性的视角执行对象从Type 1系统提升为Type 2系统，从而消除匀速斜坡输入的稳态误差。实现约束：

1. `move_integral_time_ms=0` 表示关闭，源码默认保持关闭，避免未经实测直接改变所有部署。
2. 设备限速帧不提交新积分，防止远距离或设备饱和期间wind-up。
3. 单轴误差换向时清空该轴积分，避免方向反转后旧积分继续拖拽。
4. 首轮候选原计划进入5 px稳定半径即清空全部积分；复测证明该规则会切断匀速目标所需的持续积分，现已改为仅在积分低于设备可执行阈值或误差换向后进入静止稳定，详见 `docs/018移动PI稳定态冲突修复20260714.md`。
5. 积分速度最多占设备总预算的50%，比例修正始终保留。
6. 流水线新增 `IntegralCountsX/Y` 和 `IntegralTimeSeconds`，可直接审计积分贡献和现场配置。

## 现场候选配置

仅更新CUDA NDI/UDP测试配置：

```ini
move_response_ms = 80.00
move_max_speed_cps = 1440.00
move_integral_time_ms = 320.00
pipeline_tracer_max_frames = 2000
```

响应恢复静止基线80 ms，避免同时修改响应与积分。2000帧只扩大诊断容量，不改变控制行为，用于保留完整反转数据。

## 分析器修正

相对观测速度在单向闭环中也会因检测量化和自瞄修正频繁换符号，旧算法因此把NDI单向文件误报为2～4次反转。修正后：

- 单向文件必须保持同一误差侧，不再报告反转。
- 反转由误差持续位于一侧、穿过中心、再持续进入另一侧确认。
- 反转开始使用旧误差侧最近稳态到新侧之间的极值近似。
- 恢复时间定义为从该极值到误差进入±8 px并连续保持3帧。
- `ObservedVelocityX/Y` 保留用于相对运动审计，不再单独决定真实目标反转。

## 单元测试

- 合成匀速目标以800 counts/s运行4秒：纯比例控制必须保留大于35 px稳态误差，320 ms积分候选必须将误差降到5 px以内。
- 140 px静止阶跃启用积分后必须进入5 px稳定半径，不得因积分持续输出。
- 分析器测试覆盖试次分段、持续误差侧反转、中心带恢复、counts/px恢复和场景CSV导出。
- 配置测试验证自动跟踪参数推导不会覆盖用户积分时间，新配置默认仍为0。

## 复测要求

使用新CUDA Release，在 `C:\Users\16143\Desktop\XenMovingPI\CUDA\ndi` 和 `udp` 下重新采集相同六个文件。验收顺序：

1. 单向场景 `ApproxClosedLoopLagMs` 应由约84～85 ms显著下降，目标小于30 ms。
2. NDI单向P95基线约62.6 px，目标至少下降25%至47 px以内；UDP基线约52.2 px，目标至少下降至39 px以内。
3. `SpeedLimitedPct` 应保持低位；若明显升高，再判断是否接近1440上限，不提前加速。
4. 队列最大深度继续不超过1。
5. UDP反转必须完整保留至少5次翻转；恢复次数必须等于翻转次数。
6. 移动数据通过后，使用相同积分配置执行CUDA NDI/UDP静止上中、左下、右下三点冒烟；稳定态退出必须为0，稳定后误差不得超过8 px。

若单向误差改善但反转过冲或静止冒烟失败，只增加积分时间到400～500 ms削弱积分，不修改响应时间和速度上限。只有移动与静止均通过后，才把源码默认积分时间从0推广为实测值。

# 四链路复测与最小测试矩阵

## 背景

在 `014四链路九宫格复测优化20260713.md` 将最大设备速度由1200提高到1440 counts/s后，2026-07-13 重新采集 `CUDA/DML × NDI/UDP` 四条链路的完整九宫格静止目标数据。本轮需要验证提速收益、稳定性边界，并判断后续是否仍需每次执行四条链路的完整九宫格，避免重复测试长期占用主线时间。

四份配置继续保持320×320检测区域、2560×1440完整FOV、80 ms响应、1440 counts/s、KMBOX NET和CS灵敏度一致。除推理后端、模型格式和捕获方式外，不引入其他控制变量。

## 排查方法

使用 `tools/analyze_nine_grid.ps1` 按超过100 ms的记录间隔拆分36个独立位置，并与1200 counts/s基线比较。除平均首次稳定、P95、稳定后误差、稳定态退出、远距限速、推理FPS、源到达FPS和观测年龄外，增加以下交叉判断：

1. 对相同九宫格位置比较 NDI 与 UDP，判断传输差异是否依赖推理后端。
2. 对相同传输比较 CUDA 与 DML，判断控制器行为是否依赖推理后端。
3. 使用 `首次稳定时间 / 起始误差` 辅助消除每次人工摆放位置不完全一致的影响；该指标只用于矩阵裁剪，不代替真实稳定时间。
4. 检查最坏稳定后误差是否越过8 px释放半径。5 px是首次进入稳定半径，进入后5～8 px属于控制器滞回的合法保持区间。

## 1440复测结果

| 链路 | 1200平均稳定 | 1440平均稳定 | 改善 | P95 | 远距限速 | 稳定后最大误差 | 稳定态退出 |
|---|---:|---:|---:|---:|---:|---:|---:|
| CUDA + NDI | 340.1 ms | 323.3 ms | 4.9% | 374 ms | 61.2% | 4.98 px | 0 |
| CUDA + UDP | 347.8 ms | 299.1 ms | 14.0% | 333 ms | 66.2% | 5.06 px | 0 |
| DML + NDI | 347.1 ms | 319.1 ms | 8.1% | 373 ms | 66.4% | 4.90 px | 0 |
| DML + UDP | 376.3 ms | 318.9 ms | 15.3% | 360 ms | 66.7% | 4.98 px | 0 |

四条链路均取得改善，平均首次稳定降低16.8～57.4 ms；远距限速由上一轮79.3%～84.4%降低到61.2%～66.7%。全部36段仍只进入稳定态一次，没有稳定态退出，源到达约236～239 FPS，移动队列最大深度均为1。

CUDA+UDP 左中位置稳定后最大误差为5.06 px，略高于5 px进入半径，但明显低于8 px释放半径且没有恢复输出，属于滞回设计的正常保持区间。该结果说明1440仍稳定，但近中心裕量已经需要通过重复采样确认，不具备继续提高速度上限的数据条件。因此保持 `move_response_ms=80` 和 `move_max_speed_cps=1440` 不变。

## 后端与传输关系

| 链路 | 平均起始误差 | 平均稳定时间 | 平均 ms/px | 推理FPS | 平均观测年龄 |
|---|---:|---:|---:|---:|---:|
| CUDA + NDI | 120.7 px | 323.3 ms | 2.751 | 140.9 | 8.5 ms |
| CUDA + UDP | 123.4 px | 299.1 ms | 2.481 | 129.2 | 8.2 ms |
| DML + NDI | 120.8 px | 319.1 ms | 2.688 | 95.1 | 12.7 ms |
| DML + UDP | 122.5 px | 318.9 ms | 2.648 | 95.2 | 13.7 ms |

归一化结果集中在2.48～2.75 ms/px，四条链路的控制闭环没有结构性差异。DML的NDI与UDP平均稳定时间只差0.2 ms；DML推理约95 FPS，较容易由推理吞吐掩盖传输链路的小差异。CUDA推理更快，NDI与UDP平均相差24.2 ms，同时出现本轮唯一超过5 px但未越过8 px的稳定后峰值，因此更适合作为日常双传输对比和高频稳定性压力环境。

单个位置只有一次人工采样，配对差值受起始位置、目标框量化和帧相位影响，不能把某个位置数十毫秒差异解释为稳定的后端交互。矩阵裁剪依据是四条链路的整体稳定性、归一化区间、源到达和队列结果，而不是宣称CUDA与DML在统计上完全等价。

## 最小测试矩阵

当前阶段可以减少重复测试，但区分“开发调参”和“发布验收”：

### 日常静止目标调参

- 完整测试只执行 `CUDA + NDI` 与 `CUDA + UDP` 两条链路的九宫格。
- CUDA具有更高推理吞吐，可更直接暴露NDI/UDP差异；同时对高频近中心行为更敏感。
- 两条链路必须使用同一版本程序、同一控制配置和相同九宫格顺序，不能分别调整参数。

### 发布前共享代码验收

- CUDA的NDI和UDP继续执行完整九宫格。
- DML至少使用较高观测年龄的UDP执行上中、左下、右下三个冒烟位置，覆盖垂直中距离和左右远距离；任何稳定态退出、超过8 px或明显变慢都恢复四链路完整测试。
- DML Release和CUDA Release构建及两端单元测试仍是强制项，测试矩阵裁剪不改变项目双后端构建铁律。

### 必须恢复四链路完整九宫格的条件

- 修改DML/CUDA推理、预处理、后处理、检测框坐标或模型。
- 修改NDI/UDP取帧、源尺寸、时间戳、最新帧队列或观测年龄计算。
- 修改控制器、滤波、稳定半径、释放半径、响应时间、速度上限或输入发送队列。
- 单后端冒烟出现稳定态退出、稳定后误差超过8 px、队列深度大于1，或平均稳定时间相对本轮基线退化超过10%。
- 准备正式发布或更换显卡、驱动、模型、输入设备、游戏灵敏度及发送端编码配置。

## 结论与建议

目前不需要每轮都重复四条链路的完整九宫格。日常参数迭代可只在CUDA环境下完成NDI和UDP完整测试；DML保留发布前UDP三点冒烟和强制Release构建。若工作内容属于后端、捕获、坐标或控制器共享逻辑变更，则恢复四链路完整九宫格。

下一轮提高数据可信度的优先事项不是继续加速，而是对CUDA+NDI和CUDA+UDP各重复三次九宫格，使用中位数和P95判断24 ms传输差异是否稳定存在。该重复性验证记录为非阻塞待办，不影响当前1440参数闭环。

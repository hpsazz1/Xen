# 320与240毫秒积分对照

## 背景

2026-07-14 收到两组最新 DML+NDI 持续移动与高频反转数据：

- `C:\Users\16143\Desktop\XenMovingPI320\DML\ndi`
- `C:\Users\16143\Desktop\XenMovingPI240\DML\ndi`

用户确认六份 CSV 的 `BuildRevision=36ea169bdf80-dirty` 是本轮最终验收基线；后续仅更新文档，没有重新构建，因此本轮不以当前仓库 `HEAD` 重新判定程序身份。

两组 CSV 均为 `BuildBackend=DML`、`ControllerRevision=4`、`move_response_ms=80`、`move_max_speed_cps=1440`，唯一配置变量是 `move_integral_time_ms`：320 ms 或 240 ms。

## 分析器修复

发现高频反转分析器在确认一次方向后没有清空 `candidateDirection`。下一帧会继续沿用旧方向，同时使用 `candidateStart=-1` 参与索引，造成反转起点回跳并输出负恢复时间。

已修复 `tools/analyze_moving_target.ps1`，并增加恢复时间非负和场景恢复 P95 透传测试。场景级 `RecoveryP95Ms` 取各试次最大值，避免最差反转试次被平均掉。

## 结果对比

| 场景 | 指标 | 320 ms | 240 ms | 变化与判断 |
|---|---|---:|---:|---|
| 持续向左 | 末尾1秒 P95 | 3.19 px | 1.55 px | 240 改善 51.4% |
| 持续向右 | 末尾1秒 P95 | 3.50 px | 2.86 px | 240 改善 18.3% |
| 高频反转 | 整段 P95 | 19.39 px | 17.64 px | 240 改善 9.0%，未达到10%门槛 |
| 高频反转 | 恢复均值 | 63.4 ms | 74.6 ms | 240 变差 17.7% |
| 高频反转 | 恢复 P95 | 79 ms | 108 ms | 240 变差 36.7% |

两组数据的单向试次起始误差不一致，例如右移试次起始约 `-0.4/+57.5/+80.5 px`（320 ms）与 `+82.0/+134.5/+101.0 px`（240 ms），因此整段 P95 主要受追赶阶段影响，不能据此否定 240 的稳态改善；但高频反转持续约25/29次、恢复指标在同一脚本口径下直接恶化，足以否定其作为当前现场默认值。

## 优化结论

1. 保持现场推荐 `move_integral_time_ms=320`，不修改源码默认值、响应时间 `80 ms` 或速度上限 `1440 counts/s`。
2. 保留分析器修复，后续所有高频数据必须检查 `RecoveryMeanMs>=0`、`RecoveryP95Ms` 和 `RecoveredReversals=ReversalCount`。
3. 下一轮若继续优化，只做固定起始误差的 `280 ms` 单变量候选测试，并同时完成静止冒烟；不得根据本轮不同起点的整段 P95 直接推广 240 ms。

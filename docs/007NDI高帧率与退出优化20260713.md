# NDI 高帧率与退出优化（2026-07-13）

## 背景

OBS 视频设置为 240 FPS，并将 2560×1440 主画面按 `x=1120`、`y=560`、`320×320` 发送。接收端 CSV 的捕获 FPS 约为 132～155，但旧诊断只有一个 `FPS`，无法判断 OBS 是否真的在 NDI 帧头声明 240 FPS、网络实际收到多少帧、辅机消费多少帧以及检测器发布多少结果。同时启动时出现两轮“source not found / Starting”，关闭控制台后进程也不能立即按统一流程退出。

## 技术选型

NDI 官方文档说明视频帧的 `frame_rate_N/frame_rate_D` 是发送端声明帧率，不能用游戏 FPS 或 OBS 设置值代替。NDI Tools Screen Capture HX 使用 GPU 加速，官方描述支持 4K、120 Hz 或更高，但不承诺所有设备和模式稳定达到 240 FPS；它主要面向完整桌面采集，不能替代 OBS 对 320×320 中心 ROI 的精确裁剪。因此当前优先保留 OBS 低带宽 ROI，并用真实分层统计定位瓶颈。NDI Tools 可作为 A/B 对照源，而不是默认替换方案。

选型依据：[NDI 视频帧字段](https://docs.ndi.video/all/developing-with-ndi/sdk/frame-types)、[NDI Screen Capture HX](https://docs.ndi.video/all/using-ndi/ndi-tools/ndi-tools-for-windows/screen-capture-hx)、[NDI Analysis 接收帧率说明](https://docs.ndi.video/all/using-ndi/utilities/analysis)。

## 原因

启动重复日志来自两点：捕获线程在进入循环前已经创建 NDI 后端，而主程序又预置 `detection_resolution_changed=true`，第一轮循环立即销毁并重建同一后端；每个新 finder 又在 mDNS 首轮结果返回前同步读取，因此打印误导性的“指定源不存在”。

控制台关闭延迟来自缺少 `SetConsoleCtrlHandler`。点击窗口关闭按钮没有进入现有的 `RequestApplicationShutdown` 协调路径，线程和设备只能依赖系统结束进程。KMBOX 代码中的 `kmNet_monitor(10000)` 参数实际是监听端口 10000，不是 10 秒超时；正常设备停止命令的网络等待上限约为 300 ms。

## 实现

- 移除启动前多余的检测分辨率变更标志，避免 NDI 后端启动即重建。
- 初始源连接统一交给 NDI 接收线程，后台等待 mDNS 首轮结果，不再把正常发现延迟打印成错误。
- 读取每个 NDI 视频帧的 `frame_rate_N/frame_rate_D`，统计实际接收 FPS、编码帧尺寸、累计接收帧和接收队列丢帧。
- DetectionBuffer 统计检测结果实际发布 FPS。
- 采集详情 UI 和流水线 CSV 同时输出发送声明、NDI 接收、捕获处理、检测发布四级 FPS。
- 注册控制台关闭、Ctrl+C、注销和关机处理器；先请求统一关机，关闭窗口事件最多等待 4 秒让主线程完成 join 和设备清理。

## 单元测试与验收

单元测试检查 CSV 新诊断字段。提交前执行 `git diff --check`、基础算法测试、DML Release 和 CUDA Release 构建。

实机复测时按以下顺序判断：

1. `NdiDeclaredFPS`：OBS/NDI 插件实际声明的帧率；若不是 240，瓶颈在发送端或插件。
2. `NdiReceiveFPS`：网络与解码后实际收到的帧率；与声明值的差距反映发送、网络或解码丢失。
3. `FPS`：捕获线程消费率；低于接收率且 `NdiDroppedFrames` 增长说明辅机消费不足。
4. `InferenceFPS`：检测结果发布率；低于捕获率说明 DML/TRT 推理是主要瓶颈。

使用 NDI Tools 做 A/B 测试时必须保持相同分辨率、帧率、编码模式和网卡条件。Screen Capture HX 若只能发送完整桌面，其额外编码像素和 HX 编解码延迟不能与 OBS 320×320 ROI 直接横向比较。

# 中文MSVC头文件依赖与陈旧构建修复

## 背景与现场数据

用户按r48交付版本重新完成left/right各三轮DML、NDI复测，仍感知轻微复位。新CSV的`BuildRevision`为`9692e4e25242`，但六轮全部记录`ControllerRevision=47`，与预期r48矛盾。分析器统计200 ms热身后两侧预测中断、预测方向翻转和X/Y最终输出换向均为0，最终输出同方向步进P95均为3 counts；left/right真实稳态误差P95三轮均值分别为11.5/14.0 px，脉冲频率分别为3.35/1.77/1.33 Hz和3.68/2.22/2.05 Hz。

left在热身期仍有三个自运动抑制空洞：第一轮122 ms和166 ms各中断一次，第二轮120 ms中断一次，均表现为已有保持被持续运动锁存穿透。这与r47状态机完全一致，因此当前数据只能证明r47控制周期优化仍有效，不能用于评价r48修复。

## 排查过程与根因

构建目录中的`CMakeFiles/rules.ninja`把MSVC依赖前缀缓存为乱码`娉ㄦ剰: 鍖呭惈鏂囦欢:`，而编译器实际输出为`注意: 包含文件:`。Ninja据此未能从`/showIncludes`输出生成头文件依赖，修改`target_predictor.h`后不会自动重编译其主要消费者`mouse.cpp`。

对象时间进一步闭环：旧构建中`pipeline_tracer.cpp.obj`和`mouse.cpp.obj`仍早于r48修改，而`build_identity.cpp.obj`因程序提交身份变化被重编译。最终产物于是同时包含新`BuildRevision`和旧`ControllerRevision=47`/旧预测状态机，形成“版本号更新、算法对象未更新”的假新版本。单纯核对可执行文件中的Git提交字符串无法发现这种混合对象问题。

曾验证通过设置`VSLANG=1033`并重新生成CMake缓存来恢复英文包含提示，但当前VS环境仍输出中文，不能作为可靠修复；强行匹配一个本机前缀也会绑定区域设置。根因属于构建工具本地化兼容性，继续调整预测阈值会污染算法结论，因此本轮不修改r48控制逻辑和参数。

## 设计与实现

所有项目正式包装构建统一通过`New-CMakeApplicationBuildArguments`生成命令，并追加`--clean-first`：

```text
cmake --build build/<backend> --config Release --parallel --clean-first
cmake --build build/<backend> --config Release --target ai --parallel --clean-first
```

这会在每次包装构建前清除当前CMake应用输出并重编译全部应用消费者，使算法对象、流水线CSV和`BuildIdentity`来自同一次编译。外部OpenCV位于独立安装/构建目录，不属于当前应用构建树，因此不会被重新下载或重编译。方案牺牲少量应用增量构建时间，换取在所有VS语言环境下都成立的产物完整性不变量，复杂度和维护风险最低。

构建脚本单测固定校验`--clean-first`参数；算法测试额外解析流水线首行前五列，要求CSV中的`ControllerRevision`为48。后者直接覆盖此前遗漏的风险：不再只验证UI身份常量，还验证实际编入`pipeline_tracer.cpp`对象的修订号。

20260721复现启动闪退时，Windows事件日志定位到`Config::game_profiles`的`unordered_map::clear()`访问冲突；原因是中文`/showIncludes`前缀持续乱码，增量构建保留了旧`Config`布局对象。除包装器默认`--clean-first`外，`Import-VisualStudioEnvironment`现在固定设置`VSLANG=1033`，使CMake/Ninja能记录真实头文件依赖，避免后续增量构建再次产生跨对象布局不一致。修复后DML/CUDA全量重建并启动保持运行，相关回归测试各3/3通过。

## 验证与复测要求

程序提交`f7c3b0d737fe`完成后，正式DML/CUDA包装脚本分别清理并重编译86/92个目标，两套完整CTest均为9/9通过，构建脚本单测通过。DML/CUDA构建定义分别记录`DML`/`CUDA`后端、统一程序修订`f7c3b0d737fe`和构建时间`20260716T094314Z`/`20260716T094339Z`；两个Release可执行文件均包含该程序修订且不包含`f7c3b0d737fe-dirty`。算法测试生成流水线行并确认第5列`ControllerRevision=48`，证明流水线对象、UI身份源和r48算法消费者来自本轮干净编译。

`git diff --check`通过。构建只保留CUDA SDK头文件既有C4819编码警告和历史`x64`旧可执行文件提醒，正式产物为`build/dml/Release/Xen.exe`与`build/cuda/Release/Xen.exe`。

新DML产物交付后，先采集数秒CSV并确认`BuildRevision`等于本次程序提交且`ControllerRevision=48`，再完成left/right各三轮正式复测。验收线保持不变：200 ms热身后预测中断和X/Y输出换向均为0，输出步进P95不高于3 counts，真实稳态P95不高于r47的13.17/12.83 px。若有效r48仍有复位，再基于对应帧的门控字段继续定位，不预先堆叠算法补丁。

# 透视坐标修正与jump视野逃逸优化

## 背景与数据身份

2026-07-14使用 `BuildRevision=52bd75877409`、`ControllerRevision=9`、DML+NDI、50 ms前瞻、50 ms速度窗口、1.0强度和2000 counts/s完成第三轮实测。left/right预测生效率达到99.8%/99.9%，中断次数降至3/1，中位连续运行达到244/302帧，证明r9连续性修复有效。

仍存在两类问题：static X/Y预测生效率反弹到8.5%/16.6%，Y提前P95达到12.94 px；horizontal_jump仅采集658行并被切成8段，其中有效段只有0.86~3.31秒，目标跳动后频繁丢失。测试期间为尝试重捕获还手动移动了鼠标，因此jump数据只能用于定位逃逸边界，不能用于标准预测参数比较。

## jump丢失链路

目标丢失前的最后观测多数位于320裁剪边缘：`RawPivotX` 约3或315，检测框已经贴住0或300，X误差最大达到约±258 px，且多数帧触发2000 counts/s限速。目标离开320×320传输画面后，检测器没有任何像素可供识别；按安全规则，SOT滑行只保留锁定身份而不生成鼠标控制，预测器也在真实目标丢失后重置。因此不能靠“丢失后继续盲追”解决，必须在目标到达裁剪边缘前提高真实追赶能力。

## 共同坐标根因

程序此前把完整FOV按 `fovDegrees/pixelSpan` 线性换算像素角度，但游戏画面是针孔透视投影。对于106°水平FOV和2560像素源宽：

- 旧线性换算得到约1.344 counts/px。
- 透视投影中心真实值约1.929 counts/px，相差43.5%。

控制链使用旧值时，同样像素误差发送的设备计数少约30%，降低追赶速度；自身视角补偿反向换算时又把程序移动造成的画面位移高估约43%，使静止目标在准星收敛过程中产生虚假的补偿后速度。两个现象来自同一坐标模型错误，不应继续分别调整预测门控和速度参数。

## r10设计

1. 在 `AimCoordinateSpace` 中统一实现中心透视映射：`pixel -> atan -> angle` 和 `angle -> tan -> pixel`，控制输出、运动补偿和视频回放使用同一坐标来源。
2. 检测仍是完整源画面的中心1:1裁剪，透视公式使用2560×1440源尺寸，不使用320检测尺寸；裁剪只限制可见范围，不改变单像素投影比例。
3. `move_max_speed_cps` UI和配置上限从2000扩展到4000，默认值不变。根据jump实测速度，下一轮使用3200 counts/s：透视换算后中心速度约1659 px/s，高于jump预测速度P95约1251 px/s，并留有追赶误差余量。
4. 不增加目标丢失后的盲追控制。目标飞出、死亡或角色死亡仍立即停止预测和输出。
5. 手动鼠标输入不属于程序已发送的移动，当前无法在所有WIN32、KMBOX和外部设备路径上统一取得可靠时间戳；标准数据采集期间禁止手动辅助，否则会被观测为额外视角运动并污染预测评估。

控制器修订号升级为10。单元测试固定106°/2560/1.4/0.022的透视换算结果，验证像素角度双向回算、中心裁剪不得替代源尺寸以及UI版本身份。

## 下一轮测试

UI设置 `move_max_speed_cps=3200`，预测仍保持 `50/50/1.0`，全程不手动移动鼠标：

- 先采集static九段，确认坐标修正后X/Y `PredictionActivePct<=5%`、提前P95均不超过2 px，且无持续左右或上下摆动。
- left/right各4轮，要求预测生效率至少95%、中断不超过10、无预测侧翻；X轴P95不得高于r9的34.29/33.14 px。
- jump连续20秒。首要验收是形成单个至少15秒有效试次，目标贴边丢失间隔为0；其次要求X轴P95低于100 px、限速比例不高于20%。
- reverse和 `Target_static_Role_reverse` 各20秒，确认提高速度上限没有增加实际反转之外的预测翻向。
- 目标飞出、目标死亡、角色死亡各10次，确认丢失后无预测输出。

若3200仍无法防止jump离开320裁剪，下一步只能在4000安全上限内继续实测，或扩大NDI传输/检测裁剪；不得通过目标丢失后盲目发送鼠标来伪造持续跟踪。

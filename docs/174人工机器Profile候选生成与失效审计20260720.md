# 人工机器Profile候选生成与失效审计

## 背景与目标

`DML|13f67257f5c2|r64` 的三轮主动协议已获得 9 个文件、360 个有效孤立脉冲，并给出 `MANUAL_REVIEW_ONLY`。协议结果不能直接改写 `[Games]` 或启用运行时缓存。本轮增加独立候选生成器，把人工审核通过的协议、当前只读配置和明确的探针 ROI 固化为不可覆盖的 schema v1 文件，再用同一 C++ 加载器反向验证精确命中和逐字段失效。

边界保持不变：工具不修改 `config.ini`、`[Games]`、legacy 或 active，不设置 `machine_profile_cache_enabled=true`，也不把候选复制到隐式位置。

## 设计与安全门禁

正式入口为 `xen_machine_profile_candidate`。它只接受绝对路径和显式 `--confirm-manual-review YES`，并依次验证：

- decision 必须完整包含协议身份、轮次、trial、比例、时序和人工审核字段；`ProtocolPassed=1`、`Recommendation=MANUAL_REVIEW_ONLY`、`ProfileAutoWrite=0` 且 `Issues` 为空；
- 至少 3 轮、120 个 trial，summary 必须覆盖每个轮次、幅值和 X/Y 轴组合，所有行通过且 trial 总数一致；
- 支持 PowerShell 输出的 UTF-8 BOM、标准带引号 CSV；重复字段、缺列、重复组合、格式错误或失败行全部拒绝；
- 首版证据仅支持 NDI，配置后端必须同时匹配协议身份和工具构建后端，控制器修订必须等于当前 r64；
- decision、summary 和 config 分别计算 SHA-256，并合并写入 `EvidenceDigest`；写入前后再次计算 config 哈希，任何变化都删除候选并失败；
- 输出必须是不存在的绝对路径，`saveNew` 拒绝覆盖已有文件。生成后由同一 `MachineProfileCache` 反向加载，精确键必须进入 Level 3，否则删除候选。

探针 ROI 与运行时键 ROI 是两种不同坐标语义：本轮光流证据 ROI 为接收帧内 `120,100,80,80`，写入 `ProbeRoi*`；运行时检测 ROI 由 2560×1440 源和 320 检测分辨率计算为居中的 `1120,560,320,320`，写入严格缓存键 `Roi*`，不得混用。

## 逐字段失效审计

生成器在真实候选反向加载后，分别只改变一个键字段并调用正式运行时判定器。审计覆盖游戏 Profile、瞄准模式、采集方式、NDI 源、源宽高、ROI 四字段、后端、输入方式、设备身份、灵敏度、yaw、pitch、水平/垂直 FOV、FOV 缩放标志、基础 FOV 和控制器修订，共 21 项。

每项都必须满足：Level 3 不再命中、立即降为 Level 2、`cacheMatched=false`，且原因精确为 `cache_key_mismatch:<字段>`。任一项不满足都会删除刚生成的候选并返回失败。本轮真实候选 21/21 通过，因此严格失效契约成立；这仍不等价于启用后的实机 shadow 验收。

## 测试与后续

PowerShell 集成测试覆盖候选成功生成、BOM/带引号 CSV、运行时/探针 ROI 分离、配置哈希不变、不可覆盖和失败协议不产物。缓存单元测试继续覆盖 schema 读写和四级降级。提交前 DML 与 CUDA Release 均完成 clean-first 构建，两后端 CTest 均为 13/13 通过；CUDA/OpenCV 既有头文件告警未形成新增编译错误。

下一步只能由人工明确启用独立缓存路径后采集 shadow CSV，核对 UI/CSV 的 Level 3 精确命中，并在现场改变 NDI 源或其他安全字段验证 Level 2 降级；在完成该实机验收前，active 与 legacy 继续冻结。

# 自动推导覆盖移动参数修复

## 背景

九点方位复测发现，测试配置仍为 `move_max_speed_cps=960`，导出的 CSV 也缺少新版 `SpeedLimited`。进一步检查确认，旧 DML 可执行文件仍包含自动推导的 960 固定值；即便用户手动修改配置，程序启动和运行中实际 FPS 档位变化时也会再次覆盖内存值，随后可能被 UI 配置脏检测写回文件。

## 原因与调用链

1. `Config::loadConfig()` 正确读取用户配置。
2. 主程序启动后立即调用 `applyAutoDerivedTrackerParams()`。
3. 鼠标线程在检测分辨率或稳定 FPS 档位变化时重复调用同一函数。
4. 该函数除了推导目标跟踪参数，还固定写入 `move_response_ms` 和 `move_max_speed_cps`，混淆了跟踪器自动参数与设备实测标定。
5. Overlay 在自动模式下隐藏两个移动滑块，并可能把程序内部覆盖后的值延迟保存到 `config.ini`。

## 设计决策

`move_response_ms` 决定控制响应形态，`move_max_speed_cps` 表示设备每秒计数上限。控制器已经使用真实观测时间间隔把每秒速率换算为单帧预算，因此这两个值不应随检测分辨率或捕获 FPS 自动变化。

自动推导只保留确实依赖分辨率或 FPS 的目标确认、终止、滑行、测量噪声和追踪控制参数。移动响应、设备上限及运动变化保护均由用户配置和实测数据持有。

## 实现

- 从 `applyAutoDerivedTrackerParams()` 移除移动响应时间和最大设备速率写入。
- UI 将开关改名为“自动优化跟踪参数”，明确其职责范围。
- 无论自动跟踪优化是否开启，移动响应时间和最大设备速度始终可见、可编辑。
- 增加独立配置单元测试，覆盖启动推导以及运行中分辨率/FPS 变化后的重复推导。

## 验证要求

- 配置测试验证自定义的 73 ms、1337 counts/s 在两次自动推导后保持不变，同时 FPS/分辨率相关的跟踪参数确实更新。
- 基础算法测试验证1200默认速度、实际 FPS 帧预算和 `SpeedLimited` 导出字段。
- 完成 DML Release、CUDA Release 双后端构建。
- 检查新 DML 可执行文件包含 `SpeedLimited`，且生成时间晚于本次源码修改。

## 后续复测

使用新 DML 程序和 `move_max_speed_cps=1200`，按左上、左中、左下、上左、上中、上右、右上、右中、右下相同顺序复测。新 CSV 必须包含 `SpeedLimited`；否则应首先确认启动的可执行文件和同目录 `config.ini`，不进入算法调参。

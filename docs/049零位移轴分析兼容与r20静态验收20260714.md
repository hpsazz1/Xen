# 零位移轴分析兼容与r20静态验收

## 背景

r20将预测正式释放条件从三次提高到四次可靠方向确认。本轮使用`C:\Users\16143\Desktop\Xen\DML\ndi\static.csv`复测九段静止目标，验证确认边界的残余误预测是否消失。

## 数据身份与场景

- 后端：DML。
- 构建提交：`6b3b5021332f`。
- 控制器修订号：20。
- 参数：最大设备速度3200，预测提前量50 ms、速度窗口50 ms、强度1.0。
- 数据共656行，按超过150 ms的采集空档切分为九段；每段去除前200 ms预热后共481个有效样本。
- DML推理平均95.1 FPS，NDI接收平均238.3 FPS。

## 分析结果

- X/Y框外预测帧均为0，预测生效率均为0%，低于不高于0.5%的验收上限。
- 九段X轴平均P95误差为4.44 px，Y轴平均P95误差为5.11 px。
- r19残留的三个确认边界预测帧已经全部消失，证明第四次可靠方向确认命中根因。
- 相比r19的4.23/4.89 px，基础误差轻微变化0.21/0.22 px，属于不同轮次检测和操作波动，没有证据表明新增确认损害基础跟踪。

## 排查发现

正式移动分析脚本无法直接完成本轮报告。原因不是CSV损坏，而是部分static试次在被分析轴上完全没有请求位移，`countsPerPixelSamples`为空；脚本仍把空数组传入百分位函数，PowerShell在函数调用前即拒绝参数绑定。

这类零位移是稳定跟踪的正常结果。将其当作异常会使越稳定的数据越难自动验收，也会阻断后续批量对比。

## 解决方案

- 当试次存在有效请求位移时，继续使用请求counts与像素的比值中位数估算counts/px。
- 当某一轴完全没有请求位移时，`EstimatedCountsPerPixel`和由其推导的`ApproxClosedLoopLagMs`明确输出0，表示该试次不可估算，不伪造比例或滞后。
- 增加零位移轴回归测试，覆盖分析成功、比例为0和滞后为0。
- 不修改预测器、基础控制器或控制器修订号；r20 static参数已通过，继续调参只会增加真实移动释放延迟。

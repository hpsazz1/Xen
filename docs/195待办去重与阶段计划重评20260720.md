# 待办去重与阶段计划重评

## 背景

`docs/todolist.md`长期保留了从r30到r65的完整推进记录。随着P0-5、P0-6和机器Profile分支连续迭代，部分早期待办虽然已经被后续实测覆盖，仍保持未完成状态；另一些开放式研究项在候选被否决后仍没有终止条件。这会让后续执行者重复采集旧身份数据，或继续投入已经有充分反证的控制结构。

用户要求先整理待办并重新评估计划；P0-5若没有继续投入的必要，应直接跳过。此次只调整计划和文档，不修改源码、脚本、配置、资源或运行行为。

## 去重依据

### r30与P0-6E历史任务

r30的单轮right/reverse复测已经失去当前验证价值。r40～r65完成了多轮left/right、static、jump和reverse实测，控制器修订、CSV契约、相机响应和命令时轴均已变化；继续生成r30命名数据不能代表当前程序。

P0-6E要求的被动Profile、固定延迟A/B和static/jump/reverse复测已经由136～165号工作完成。r63/r64最终通过static负样本与jump/reverse正样本门禁，旧总项只是未同步勾选，不是剩余工作。

### Profile与复杂模型

被动Profile已经完成3份CSV、6个按键段的初测；主动Profile进一步完成3轮、9文件、360个有效脉冲。独立机器缓存、严格失效键和四级降级也已实现并完成失效审计，因此旧P1设计与重复性条目均可关闭。

自适应Q、一阶响应、Smith、速度前馈、积分和多种局部门控已有回放或实机反证。当前证据没有证明IMM、光流、SOT或三维弹道能解决剩余问题，引入这些结构会扩大项目而没有可验证收益，不符合“小而美”的定位。

## P0-5阶段决策

P0-5不按“通过”关闭，而按“证据充分后停止并跳过晋级”关闭。设备1800隔离物理协议通过，恢复代次1800使四物理端点共同通过域由485提升到689且原通过损失0；但left/right仍有113个高延迟、高速P95回退域，static仍有8个方向翻转域。2160只读归因仍剩50域且没有物理授权。

这些结果说明设备能力不是唯一阻断。继续扫描更高速度、响应时间、前馈、积分、Smith或static局部门控，只会重复已经完成的证伪路径。因此固定以下边界：

- 正式控制继续使用r65基线、80 ms响应、前馈0、积分0和1440 counts/s；
- P0-5保持`HOLD_SHADOW`，不进入active；
- 不再追加同类回放矩阵、设备速度协议、shadow旁路CSV或参数扫描；
- 只有真实使用暴露影响核心闭环的问题，或出现能够在实现前解释40 ms+1.5x持续单向水平相位、同时不影响static/reverse的新结构假设时，才建立新的独立任务；不得直接恢复旧P0-5。

## 重新评估后的计划

当前主线从“继续寻找P0-5晋级候选”改为“稳定版收口”。第一优先级是核对最近一次程序提交`8609041`对应的干净DML/CUDA构建、单元测试和UI/CSV/构建产物版本身份记录；已有记录完整时不机械重建，存在缺口时才使用正式脚本补齐。

稳定版收口之后只接收由真实使用触发的问题。每个问题必须包含可复现输入、实际影响和验收标准；没有真实证据的算法优化、模型扩展和参数扫描不进入主线。机器Profile继续保持人工生成、只读加载和严格失效降级，不开放自动写入。

## 验证结果

本次为纯非代码变更。检查范围仅包含`docs/todolist.md`和本记录；不需要运行单元测试、DML/CUDA构建或更新程序版本身份。提交前执行`git diff --check`并确认所有旧未完成项均已明确归并，当前只保留稳定版收口与真实使用问题回收两个可执行事项。

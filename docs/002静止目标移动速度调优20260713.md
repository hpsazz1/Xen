# 静止目标移动速度调优（2026-07-13）

## 背景

新基础移动模式已经移除多层速度曲线和轨迹整形，但左、上、右三个静止目标的实测移动仍明显偏慢。后续捕获方式、设备和双机 NDI 链路都可能变化，因此控制器不能绑定某个预设 FPS，必须始终使用捕获窗统计出的实际 FPS。

本次只调整基础控制器的设备速率上限，不重新引入预测、速度曲线或轨迹整形，保持最小闭环。

## 原因

当前控制器按以下公式限制单帧输出：

```text
单帧计数上限 = move_max_speed_cps × dt
dt = 1 / 捕获窗统计 FPS
```

原默认值为 `240 counts/s`。实测 CSV 的平均 FPS 为 131.2~155.0，因此单帧上限只有 1.55~1.83 counts。最终输出还要取整为整数，绝大多数有效帧只能发送 1~2 counts。

提高输入 FPS 不会自动提高每秒总速度：FPS 越高，单帧预算会按比例减小，每秒总预算仍锁定为 `move_max_speed_cps`。所以问题不是捕获 FPS 低，而是设备速率上限本身过低。

## 排查过程

1. 核对 `config.ini`、配置加载、自动推导、运行时配置刷新、控制器更新、整数取整和设备队列的完整调用链。
2. 确认 `auto_derive_tracker_params = true` 时，自动推导会把配置覆盖为固定的 `240 counts/s`。
3. 确认捕获线程每秒统计实际 FPS 并写入全局 `captureFps`，移动线程通过 `frameIntervalSec(captureFps.load())` 得到当前 dt；算法没有必要依赖配置的目标捕获帧率。
4. 分析 `x64/DML/右.csv`、`左.csv`、`上.csv` 的全部样本。三组数据的队列深度没有持续积压，瓶颈位于控制器限速而不是发送队列。

## 技术选型与权衡

采用成熟的时间步长控制方式：参数用 `counts/s` 表达物理设备速率，单帧预算使用捕获窗统计 FPS 对应的真实 dt。这样更换桌面复制、NDI 或其他捕获设备时，只要捕获窗能给出实际 FPS，控制器的墙钟时间速度就保持一致。

新默认值选择 `960 counts/s`：

- 来源是三组真实静止目标所需的 115~206 counts，而不是某个固定 FPS；
- 右、左、上三组主移动段理论约为 0.21、0.19、0.12 秒，符合实际游戏快速收敛需求；
- 相比原值提升 4 倍，但仍低于配置钳位上限 `2000 counts/s`，保留后续实机微调空间；
- `move_response_ms` 保持 80 ms。远距离阶段主要受设备速率上限控制，响应时间只影响接近目标后的收敛形态，当前没有数据支持同时修改两个变量。

例如 `960 counts/s` 在本轮约 131 FPS 时单帧预算约 7.33 counts，在约 155 FPS 时约 6.19 counts，在 240 FPS 时为 4 counts；三者每秒总预算都仍是 960 counts。

## 实现

1. 配置构造、缺省读取、重置和自动推导的 `move_max_speed_cps` 统一改为 `960`。
2. DML 实测配置同步改为 `960`，避免配置展示与生产默认不一致。
3. 手动调节滑块上限与配置钳位统一为 `2000`，允许在关闭自动优化后做受控实机试验。
4. 中文注释明确单帧预算来自捕获窗统计 FPS，不绑定捕获方式、设备或固定帧率。

## 单元测试

新增实际 FPS 速率一致性测试，分别输入本轮数据附近的 131、155 FPS 和未来可能出现的 240 FPS，验证：

```text
单帧预算 × 实际 FPS = 960 counts/s
```

## 后续建议

更换捕获方式或接入双机 NDI 后，重新采集同样的左、上、右静止目标 CSV，并增加一组匀速移动目标。分析时以 CSV 的实际 FPS 和时间戳为准，不以设备标称 FPS 为准。若静止目标出现越过或往复修正，优先在 `720~960 counts/s` 范围内微调最大设备速度；只有近中心收敛仍偏慢时，才单独评估 `move_response_ms`，避免一次修改多个变量导致结论不可验证。

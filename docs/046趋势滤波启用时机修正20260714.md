# 趋势滤波启用时机修正

## 背景与版本

2026-07-14使用 `BuildRevision=a76c1e4ba53d`、`ControllerRevision=17`、DML+NDI、3200 counts/s和预测50/50/1.0完成static九段复测。NDI接收平均237.9 FPS，DML推理94.4 FPS，CSV包含 `FilterTrendSpeed`。

## r17 static结果

r17在left/right移动场景收益明确，但static未通过。X/Y预测生效率为1.7%/2.1%，高于r14的0.4%/0.4%；平均P95误差为5.21/7.93 px，Y轴trial 3和9分别达到15.79/18.44 px。

逐帧检查显示误预测仍然只有1~2帧，随后会被自运动门控捕获，没有形成五帧持续运动豁免，也没有抑制结束后的长时间复发。因此r15持续运动豁免和r14重新武装门槛仍然有效。

## 根因

r17从第一帧开始使用约30 ms速度趋势控制基础滤波。在static测试中，目标本身静止，但程序正在主动旋转视角把不同起始位置的目标拉向中心。趋势滤波把这段快速收敛平滑成位置拖尾，预测器将补偿后的短时残差识别为1~2帧运动提前，随后门控才撤销。

移动场景真正需要趋势滤波的是预测已经建立后的持续跟随阶段。此时检测中心存在700~1300 px/s单帧尖峰，而真实趋势约250~310 px/s。预测建立前的初始收敛不需要该降噪路径，继续全程启用会破坏static安全项。

## r18设计

1. `BasicTargetFilter`继续实时维护速度趋势，但只有上一帧预测偏移非零时，使用趋势速度控制滤波时间常数。
2. static、目标首次出现、预测门控保持、停止和反向撤销阶段使用r16以前的瞬时速度响应，避免主动收敛拖尾。
3. 持续left/right预测建立后，从下一观测开始启用趋势滤波，保留r17对检测中心尖峰的隔离能力。
4. 新增CSV字段 `FilterTrendActive`，明确记录当前观测是否真正使用趋势响应；不能仅凭趋势速度字段猜测启用状态。
5. 不修改趋势时间常数、预测、积分、3200速度、透视换算和目标丢失逻辑。

控制器修订号升级为18。单元测试覆盖趋势模式下交替跳点抑制与持续移动响应，并新增预测建立前使用瞬时响应的回归测试。

## 下一轮测试

- 先复测static九段，要求X/Y预测生效率恢复至不高于0.5%，不得出现持续过冲；`FilterTrendActive` 在绝大多数static帧应为0。
- static通过后复测left/right各4轮，要求X轴P95不明显回退于8.29/8.88 px，预测生效率至少95%，持续预测阶段 `FilterTrendActive` 应接近预测生效率。
- 两项通过后进入jump、reverse和 `Target_static_Role_reverse` 各20秒，以及三类目标丢失各10次。

## 验证要求

执行滤波、预测与控制器专项单元测试、分析脚本回归、DML/CUDA Release构建和两套完整CTest；中文提交并推送后按干净提交重新配置双后端，核对UI、CSV和可执行文件为同一提交且 `ControllerRevision=18`。

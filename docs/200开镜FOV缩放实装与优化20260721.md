# 开镜FOV缩放实装与优化

## 背景

项目已有`fovScaled/baseFOV`配置、Profile UI和`zooming`按键状态，但控制链始终使用固定`fovX/fovY`。因此勾选FOV缩放后，按下开镜键不会改变像素到角度、角度到counts或运动补偿比例，功能只完成了配置外壳，没有形成运行闭环。

## 原因与排查

配置解析和保存支持五字段游戏Profile，键盘线程也会按`button_zoom`更新`zooming`。实际检查legacy控制、shadow角度观测、被动标定、人手角速度和自运动补偿后发现，它们都直接读取固定`fov_x/fov_y`；`Config::degToCounts`虽然接受`fovNow`，但主控制链没有调用它。Profile在UI中修改后也只标记待保存，没有刷新`MouseThread`缓存。

## 设计与实现

游戏Profile扩展为`sensitivity,yaw,pitch,fovScaled,baseFOV,scopeFOV`。`baseFOV`表示灵敏度标定时的腰射水平FOV，`scopeFOV`表示按住缩放键后的实际水平FOV。统一解析器按针孔投影使用`tan(currentFOV/2) / tan(baseFOV/2)`计算灵敏度比例，并用同一投影关系推导垂直FOV；不能继续使用旧的线性FOV比例，否则开镜前后counts/像素仍不一致。legacy、shadow、运动补偿、被动标定及人手仲裁共同使用该结果，避免链路间比例不一致。

开镜边沿会清空旧坐标系下的排队命令、滤波、预测、积分和量化余量。机器Profile缓存尚未包含scopeFOV身份，开镜期间不复用腰射标定角度比例，优先使用当前游戏Profile的动态比例。UI修改Profile后立即刷新运行时缓存。流水线CSV新增缩放开关、开镜状态、基准/开镜/有效FOV和灵敏度比例，便于实机核对。

旧五字段配置继续兼容，缺少`scopeFOV`时回退到`baseFOV`，保持升级前不缩放的行为。默认Profile仍关闭该功能，避免在不知道游戏和倍镜实际FOV时猜测参数。

## 验证要求

单元测试覆盖腰射、开镜、无效scopeFOV安全回退、配置读写和CSV字段；UI测试确认三个Profile控件可见。正式交付需完成DML/CUDA Release clean-first构建、双后端相关CTest、启动保持、r67身份和远端提交一致性核对。

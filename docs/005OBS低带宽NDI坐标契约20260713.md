# OBS 低带宽 NDI 坐标契约（2026-07-13）

## 背景

双机实测使用 OBS NDI 插件发送 320×320 画面，辅机使用 KMBOX NET 输出。流水线 CSV 的 `SourceWidth/SourceHeight` 均为 320×320，控制器因而把 ROI 当成完整 2560×1440 游戏 FOV：X 轴移动比例约放大 8 倍，Y 轴约放大 4.5 倍，造成目标附近反复越过。

直接发送 2560×1440 能保留完整尺寸语义，但像素量是 320×320 的 36 倍，会明显提高编码、内存和网络负载。负载接近瓶颈时可能产生排队和额外延迟，具体值仍须按真实 OBS/NDI 模式、帧率和网络实测。因此本次选择“小而完整”的低带宽闭环：继续发送 320×320 中心 ROI，同时独立传递完整 FOV 尺寸。

## 设计

NDI 接收端按以下优先级解析完整 FOV 尺寸：

1. 视频帧的 `<xen source_width="..." source_height="..."/>` XML 元数据；
2. 接收端 `ndi_source_width/ndi_source_height` 配置；
3. NDI 视频帧实际尺寸。

只有宽高同时有效、均不小于收到的视频帧且不超过 16384 时才接受扩展尺寸。无效或单边配置会安全回退，防止损坏元数据制造过大的移动比例。OBS 标准 NDI 插件不能稳定注入 Xen 自定义 XML，因此当前真实场景使用第 2 层；元数据支持用于后续自定义发送端兼容。

本契约只支持 1:1 像素中心裁剪。OBS 不得把完整 2560×1440 缩放、拉伸或挤压为 320×320，否则检测坐标与完整 FOV 像素不再是一一对应关系。

NDI 发现名称必须使用完整字符串。本次源名为 `HPSAZZ (OBS PGM)`，`HPSAZZ` 与括号内的 `OBS PGM` 都不能省略。游戏配置切换为 `CS = 1.4,0.022,0.022`，与实际灵敏度一致。

## 实现

- 新增无 NDI SDK 依赖的几何解析模块，负责元数据、配置回退和边界校验。
- NDI 接收线程继续处理 320×320 网络图像，但向控制器发布 2560×1440 完整 FOV 尺寸。
- 配置加载、保存、热重载和 NDI 设置界面同步支持完整 FOV 宽高。
- 单元测试覆盖元数据优先、OBS 配置回退和非法尺寸回退。

## 验证与后续采集

提交前执行 `git diff --check`、基础算法单元测试、DML Release 和 CUDA Release 构建。运行时重新采集左、右、上三个静止目标，验收条件如下：

- CSV `Resolution=320`；
- CSV `SourceWidth=2560`、`SourceHeight=1440`；
- 主轴误差越零次数和稳定区退出次数显著下降；
- 实际 FPS 继续取捕获窗统计值，不绑定 240 FPS；
- 修正坐标契约前不继续降低最大速度来掩盖根因。

后续若需要精确比较 320 ROI 与全分辨率 NDI 的延迟，应由发送端加入源帧号和单调时钟时间戳，接收端分别记录源 FPS、接收 FPS、处理 FPS及端到端帧龄。

# 孤立脉冲物理响应采集器

## 背景与结论

连续闭环 static 数据中的命令密度、控制器收敛轨迹和评价窗口共同影响拟合结果，不能辨识真实设备到画面的物理响应。为执行 `156实机物理响应辨识审计20260717.md` 冻结的采集协议，本轮新增独立工具 `xen_physical_response_probe`。它不接入 `Xen.exe` 的检测、shadow、active、按键状态或普通鼠标线程，也不修改配置和控制器参数；控制器修订保持 r63，项目仍为 `HOLD_SHADOW`。

## 调用链与设计

工具显式加载现有 `config.ini`，严格解析 `input_method` 后直接创建对应 `IMouseInput`。非法输入方式硬失败，不允许工厂静默回退 WIN32；WIN32 默认拒绝，只有额外传入 `--allow-win32 YES` 才允许。工具还要求 `--confirm-device-motion YES`，缺少确认时不会加载配置、连接设备或发送命令。

NDI 接收仍使用 `NDICapture` 的原始 SDK 视频帧和 `FrameTiming.backendReceiveTime`。普通运行继续采用只保留最新帧的低延迟队列；诊断工具显式选择 2048 帧有界顺序队列。队列覆盖、接收 FPS 低于 200、声明 FPS 低于 200、100 ms 断流或时间序列丢帧都会停止后续脉冲。原构造参数的 120 上限不控制 SDK 接收，但已放宽到 1000，避免参数含义错误地排斥 240 Hz。

用户通过检测画面坐标提供固定高对比标记 ROI。跟踪器保存首帧 ROI 模板，逐帧在上一位置前后 48 像素范围使用 OpenCV 归一化相关模板匹配；ROI 灰度标准差必须至少 15，匹配相关系数必须至少 0.75。有限搜索范围防止跳到画面远处的相似 HUD，固定首帧模板避免逐帧更新造成漂移。任一帧跟踪失败立即停止后续设备命令。该方案只依赖 DML/CUDA 两套 OpenCV 均包含的 `imgproc` 模块。

每轮先采集至少 300 ms 基线，记录 `steady_clock` 的 move 调用前时刻，调用一次孤立 counts 命令，再记录接口成功返回时刻，随后采集至少 500 ms 尾部和至少 500 ms 无命令间隔。两者只能解释为“调用尝试”和“设备接口确认”，不能宣称是硬件真实执行时刻。正负方向交替，X/Y 分轴，默认每方向 10 次、16 counts。

## 使用方法

先在固定测试场景中保持角色、目标和相机静止，确认 NDI 发送端实际输出 240 Hz。ROI 坐标基于配置中的正方形 `detection_resolution` 画面，选取固定标记内部包含多个清晰角点的区域，避免准星动画、HUD 动画、角色模型和会闪烁的边缘。

示例：

```powershell
build\dml\tools\Release\xen_physical_response_probe.exe `
  --config C:\Users\16143\Desktop\Xen\DML\ndi\config.ini `
  --ndi-source "精确的NDI源名称" `
  --output-dir C:\Users\16143\Desktop\Xen\DML\ndi\physical_probe_r63 `
  --roi-x 120 --roi-y 100 --roi-width 80 --roi-height 80 `
  --axis both --counts 16 --repeats 10 `
  --baseline-ms 300 --tail-ms 500 --interval-ms 500 `
  --confirm-device-motion YES
```

运行期间不要启动 `Xen.exe`，不要按追踪键，也不要移动鼠标、角色或目标。工具不会发送额外“回正”清理命令；正负脉冲本身按独立轮次交替。按 Ctrl+C 可停止，已完成轮次仍写入输出。

## 输出与验收

`probe_frames.csv` 保存构建身份、控制器修订、轮次、轴、signed counts、原始 NDI 帧本机接收纳秒、命令调用前/成功返回纳秒、相对 X/Y 像素位移、跟踪质量和有效性。`probe_summary.csv` 保存每轮基线/尾部样本数、最终位移及 `t10/t50/t90/t99`。分位时刻由相对本轮基线和尾部最终位移归一化后提取，不预设线性响应核；不足 1 像素、样本不足或未达到全部分位时整轮无效，并立即停止后续脉冲。

实机数据返回后必须先检查：所有轮次无丢帧且样本数符合 240 Hz 量级；正负 X、正负 Y 各至少 10 个有效轮次；分别报告各组中位数和离散度；确认正负方向符号和最终位移一致。只有跨方向、跨轴、跨轮结果稳定且可重放解释画面轨迹，才进入运行时响应模型复评。本轮不写回任何拟合参数。

## 测试记录

单元测试覆盖有界逐帧队列的顺序与容量丢失计数、合成 240 Hz 阶跃的 `t10/t50/t90/t99`、OpenCV 归一化模板匹配对 4/-3 像素已知平移的恢复，以及缺少设备运动确认时的启动硬失败。

提交前使用正式脚本完成 DML 和 CUDA Release `--clean-first` 全量构建，两个构建目录的 CTest 均为 9/9 通过。DML 工具目标和 CUDA 工具目标均成功链接；缺少 `--confirm-device-motion YES` 的启动检查在连接 NDI 和设备前硬失败。最初选择的 LK 光流在 DML 完整 OpenCV 中可用，但 CUDA 精简 OpenCV 未构建 CPU `video` 模块，链接时缺少 `calcOpticalFlowPyrLK`。最终改用两端现有 `imgproc` 都支持的归一化模板匹配，避免为单个诊断工具扩大 CUDA OpenCV 构建和运行依赖。

提交、推送、远端引用及提交后干净构建身份在完成后核对；最终产物必须显示本次提交、DML/CUDA 后端和 r63，不得包含 `-dirty`。

# P0-5设备速率二阶段确认NDI隔离

## 背景与实测现象

目标设备使用`DML|c07d9ec19068|20260720T092933Z|r65`和计划`373a40f7693e4d15ba9b192001bf26ed`执行修正后的物理协议。1440正负方向均通过，用户按要求输入`ENTER-1800:373a40f7693e4d15ba9b192001bf26ed`后，1800正方向也通过；执行器随后以`Issues=ndi_dropped_frames`停止，没有发送1800负方向命令。结果共3条summary、48条成功命令，符合在trial 3完成后触发丢帧门禁的控制流。

三个已完成trial的最低跟踪质量均不低于0.891345，命令最大调度抖动不超过0.0941 ms，比例为0.500000、0.520833和0.516667 px/count，峰值为25、25和31 px，停止距离为6、0和0 px，最终残差与串轴泄漏均为0。由此排除设备命令失败、调度超限、跟踪失效、比例异常、停止距离、回零残差和串轴泄漏；1800正方向本身通过。

## 原因与排查

执行器使用`PreserveAllBounded`逐帧NDI队列，容量为2048。旧控制流在1440通过后直接阻塞于`std::getline`等待人工计划ID，而NDI后台线程仍持续接收，主线程不消费任何帧。人工确认时间没有协议上限，约8.5秒的240 Hz输入即可填满2048帧队列；实测trial 3记录2662个样本，远高于2.566667秒在240 Hz下约616帧，证明确认后先快速消费了积压旧帧。队列满时NDI按设计累计丢帧，trial 3结束后的零丢帧门禁因此输出`HOLD_DIAGNOSTIC`并阻止trial 4。

该失败不能通过扩大队列、清零丢帧计数或要求用户更快输入来规避。扩大队列仍保留旧帧并使1800使用确认前画面；直接清计数会隐藏真实丢帧；人工计划确认本身没有可靠的最大耗时。

## 设计与实现

执行器新增统一`startStableCapture`入口：创建指定NDI源的240 Hz逐帧会话，最多等待10秒首帧，再连续执行2秒稳定性检查，最后要求声明帧率和接收帧率均至少200且丢帧为0。初次启动和二阶段重连复用同一入口，避免门禁分叉。

1440对照通过后，执行器先析构当前NDI会话并清空其队列和全局诊断，再显示并阻塞等待`ENTER-1800:<PlanId>`。确认准确后才创建全新NDI会话并重新走完整稳定性门禁；trial 3和trial 4只消费确认后的新鲜帧。重连、稳定性、原生帧率或零丢帧任一失败均写入`ndi_restart:*`并保持`HOLD_DIAGNOSTIC`，不会发送1800命令。

本修正不改变64行计划、1440/1800命令序列、第二阶段确认词、ROI、12 px停止距离上限、48 px现场偏移上限、100 ms视觉响应延迟上限、配置自动写入关闭或active自动启用关闭。

## 测试与下一步

计划回归测试新增执行器控制流契约，要求`capture.reset()`严格早于阻塞确认，重建`startStableCapture`严格晚于确认，并保留丢帧复核。既有算法测试继续覆盖视觉延迟停止锚点、真实停止距离超限和最终残差负例；离线分析测试继续覆盖完整四trial与跨速率门禁。

完成双后端Release clean-first、完整CTest、干净构建身份、中文提交和远端推送后生成新的跨设备包。下一轮必须重新生成绑定新提交的计划，不得沿用`c07d9ec`执行器或计划。现场仍先执行1440；人工确认后会看到NDI重新连接和约2秒稳定性检查，之后才开始1800。全部四trial通过也仍只允许`MANUAL_REVIEW_ONLY`，不得自动修改配置、正式1440上限或active。

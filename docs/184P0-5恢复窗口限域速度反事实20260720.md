# P0-5恢复窗口限域速度反事实

## 背景

183号实机重捕获重评确认，约300 counts的left/right恢复事件全部会在同一reset代次内退出1440 counts/s限速，实际退出帧P95为38、最大为45。该证据只解除“20帧内未退出”的假阴性，不授权提高正式设备上限。本轮按既定边界只建立离线反事实：在179号已提交端点守卫基线上，评估检测恢复代次实际退出前的有限速度包络，并重新证明四端点共同485项零损失。

## 设计与边界

命令行新增`--candidate-recovery-max-cps`，默认0。只有显式给出高于正式候选上限的有限值时才运行专用反事实；该入口不进入配置、UI、shadow或active。

每个场景和物理域先按179号参数运行基线，记录完整逐帧检测时间线；反事实严格复用该时间线。状态机遵循以下边界：

1. 仅`left/right`在一次已检测代次发生丢失后武装恢复窗口；初始连续检测段、`static`、`jump`和`reverse`结构上禁用。
2. 恢复首帧开始临时使用有限候选上限。每帧仍计算1440 counts/s对应的正式二维帧限额；当无限制请求首次不再超过该限额时，退出帧本身及后续帧立即恢复1440。
3. 若自然退出前再次丢失，当前窗口记为`InterruptedWindows`并清理；下一次恢复建立新代次。轨迹结束仍限速则记为`OpenWindows`，不得虚构退出。
4. 逐帧CSV记录恢复代次、窗口、实际启用、有效上限；矩阵和专用summary记录启用帧数以及自然退出、再次丢失、右删失窗口数。
5. 基线与反事实的检测帧数、candidate样本数和全部legacy指标必须相同；否则`CohortStable=0`，结果无效。

## 单元测试

合成回归覆盖：大残差恢复窗口确实启用并自然退出；退出后锁回1440；自然退出前再次丢失会中断并清理；初始连续检测代次不会启用；`static`即使跨越相同丢失边界也保持请求逐值相同；专用summary公开上限、启用帧和三类退出统计。默认关闭仍复现原控制结果。

## 候选选择

先在14/0端点比较两个有限候选：

| 恢复上限 | 基线通过 | 反事实通过 | 原通过损失 | 新增通过 |
| ---: | ---: | ---: | ---: | ---: |
| 1800 counts/s | 485 | 690 | 0 | 205 |
| 2160 counts/s | 485 | 760 | 0 | 275 |

两者都满足零损失，且启用帧只存在于left/right。按最小充分原则冻结较保守的1800 counts/s进入四端点验证，不因2160的更高总通过数扩大速度包络。

## 四端点结果

统一参数为候选响应80 ms、正式上限1440 counts/s、恢复上限1800 counts/s、门控CA jerk 8000 deg/s^3、阈值12 deg/s、保持120 ms、前馈0、端点守卫开启；只改变物理命令中心/响应宽度。

| 物理中心/宽度 | 179基线 | 1800反事实 | 原通过损失 | 新增通过 | 队列漂移 |
| --- | ---: | ---: | ---: | ---: | ---: |
| 14/0 ms | 485 | 690 | 0 | 205 | 0 |
| 14/2 ms | 488 | 698 | 0 | 210 | 0 |
| 15/0 ms | 485 | 690 | 0 | 205 | 0 |
| 15/2 ms | 485 | 690 | 0 | 205 | 0 |

四端点共同通过域由485增至689，原485项损失为0。共同域构成为jump 162、left 104、reverse 162、right 107、static 154；相对179号共同域新增left 99和right 105。四端点的`jump/reverse/static`核心候选指标逐值差异均为0，证明反事实没有进入非目标场景。

最终四端点合计启用帧分别为13155、12991、13155、13155；自然退出窗口均为1458，再次丢失中断窗口分别为4、10、4、4，`OpenWindows`均为0。也就是说，本轮没有把轨迹结束仍限速的右删失代次计为自然退出。

正式输出位于`build/analysis-p05-physical14x0-recovery1800-r64`及其14x2、15x0、15x2对应目录；2160筛选输出位于`build/analysis-p05-physical14x0-recovery2160-r64`。主基线仍返回`HOLD_SHADOW`和退出码3，专用decision另行记录反事实通过数、队列漂移和`RecoverySpeedCounterfactualPromotion=HOLD_SHADOW`。

## 结论与下一步

1800 counts/s恢复限域反事实通过179号四端点共同485项零损失门禁，并把共同left/right通过域由5/2提高到104/107。该结果证明“仅恢复代次、仅实际限速退出前”的有限执行包络是有效离线方向，但不等于设备上限、运行时策略或发布授权。

P0-5继续`HOLD_SHADOW`：完整810域仍未通过，且本轮没有建立实机1800 counts/s的设备能力、过冲和退出行为证据。下一步只允许设计shadow-only、零设备输出的恢复速度建议诊断，使用实机CSV计算“若上限为1800”的请求裁剪与预计退出，不修改正式1440、配置、UI或active；在该诊断完成前不得向设备发送更高速度命令。

# 物理响应与 Profile 标定阶段结论

## 背景

本阶段为确认本机 KMBOX_NET、DML、NDI 链路的实际动作比例和响应时间，先完成孤立脉冲采集器及参数扫描，再使用被动 `profile_calibration` 做运行态交叉验证。项目继续保持 `HOLD_SHADOW`，控制器修订保持 r63。

## 排查与修复

独立采集器最初把 KMBOX_NET 异步连接误判为失败，随后又暴露冷启动首条非零命令被设备 ACK 但没有动作的问题。最终实现等待监听就绪，并在测量前使用 `+1/-1` counts 最小配对脉冲预热 HID 动作通道。修复提交为 `52096c1e775f`，DML/CUDA Release clean-first 构建和两端 CTest 均通过，远端 `origin/master` 与本地一致。

## 物理响应结论

孤立脉冲扫描与冷启动冒烟测试确认：

- 8 counts 低于当前 1 px 跟踪有效门槛，不作为拟合输入；
- 16 counts 约产生 8 px 位移；
- 32 counts 约产生 16～17 px 位移；
- 64 counts 约产生 33 px 位移；
- counts 到画面位移近似为 `0.5 px/count`；
- `t50` 约 14 ms，`t90` 约 15 ms；
- 32 counts 的少量 `t99` 长尾来自最终 1 px 模板量化，不代表完整设备响应延迟。

## 被动 Profile 标定结论

多轮被动标定中，部分快照的 X 或 Y 比例接近 `0.5 px/count`，但重复性和双轴同时有效性不足：X 曾在约 `0.206～0.526 px/count` 间变化，Y 约在 `0.318～0.508 px/count` 间变化，延迟候选约 `4～62 ms`，综合可信度最高仍不足 60%。增加样本和四方向、斜对角激励后，仍存在某轴无效或双轴有效窗口不重合。

因此被动标定结果只保留为诊断证据，不写入 `[Games]`，不覆盖灵敏度、yaw/pitch、相机响应时间或控制器参数。正式配置恢复并保持：

```ini
profile_calibration_enabled = false
```

## 更新计划

1. 冻结物理响应基线为约 `0.5 px/count`、`t50≈14 ms`、`t90≈15 ms`，仅作为当前设备和链路的审计依据。
2. 停止继续追加相同方式的被动 Profile 样本，不采用单轮最优值或跨轮简单平均。
3. 保持当前正式 Profile、r63 控制器、shadow 模式及既有运行参数不变。
4. 下一步执行一次运行态 shadow/static 验证，检查暂停零命令、九点运行段、收敛态侵入和既有 r63 行为是否回退。
5. 只有未来重新设计严格的分轴主动标定协议，并通过跨轮、跨轴重复性门槛后，才重新评估机器标定结果进入 Profile。

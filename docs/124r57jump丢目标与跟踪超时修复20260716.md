# r57 jump丢目标与跟踪超时修复

## 数据身份与现象

2026-07-16复核用户上传到`C:\Users\16143\Desktop\Xen\DML\ndi`的`config.ini`和三份独立jump CSV。数据身份统一为`DML / b0ce197cb6da / r57 / legacy`，配置为320裁剪、响应120 ms、基础速度3200、条件速度4000、积分500 ms、预测50/50/1.0、跟踪器和自动参数推导开启。

三轮数据分别为167、771、625个有效观测，持续2.456、11.464、9.019秒。真实水平误差P95为117.5、95.5、81 px，均高于r56/3200四轮均值66.88 px；框内率仅35.3%、39.3%、39.0%。用户观察到的“丢目标后跟不上”与量化误差一致。

## 排查过程与根因

目标框水平触碰320裁剪边缘的帧占比只有2.99%、0.26%、0.16%，因此主要故障不是目标持续跑出检测区域。DML推理结果发布中位数约68 FPS，正常周期约15 ms，但三轮相邻有效观测存在25～48 ms调度抖动。

控制线程的`trackerStaleTimeoutMs()`却使用捕获处理FPS计算超时，实际门槛仅25～30 ms。该时钟域与跟踪器真实步进源不一致：跟踪器只在`DetectionBuffer`发布新检测结果时推进，而不是每收到一个捕获帧就推进。当一次正常推理发布间隔超过错误门槛时，控制线程会调用`resetActiveTarget()`，连带清空基础滤波、预测方向、PI、条件速度预算和整数余量。

逐帧验证显示，三轮分别有22、99、78次在25 ms以上间隔后由成熟状态直接变为`PredictionDirectionLocked=0`，合计199次硬重置。条件速度预算因预测不断冷启动，仅命中0、5、2帧；这解释了为什么新增4000条件上限几乎没有机会生效。现象属于正常推理抖动被误判为目标过期，不是继续提高速度上限可以解决的问题。

## 顶层修复

跟踪生命周期统一改用实际检测结果发布FPS：

1. 自动跟踪参数按`DetectionBuffer::getPublishFps()`推导，发布率尚未建立时才回退捕获处理FPS。这样`ml_termination_frames`和`ml_coast_frames`表示稳定的检测结果周期，不再随更快的捕获链路错误换算。
2. 无检测发布的完整状态清除门槛改为`clamp(4000 / publishFps, 100, 250) ms`。68 FPS以及更高发布率均使用100 ms，低发布率至少保留四个发布周期。
3. 最短100 ms与`TargetPredictor`允许的最大真实观测间隔一致。超过该边界后预测器本身仍会安全重建，不会复用无限期陈旧速度。
4. 正常空检测结果仍会发布新版本、推进MOT/SOT滑行并按既有`ml_coast_frames`掉锁；推理完全停发期间没有`hasAimObservation`，因此保护窗口只保留内部状态，不会凭空发送鼠标命令。
5. 流水线CSV在`InferenceFPS`后新增`TrackerStaleTimeoutMs`，复测时可直接核对实际发布率与过期门槛。控制器修订号升级为r58。

本轮不调整条件速度预算的60 px/1280 px/s进入阈值、32 px/640 px/s退出阈值、120 ms持续上限或200 ms冷却，也不调整预测、响应、积分、检测置信度和SOT重捕获参数。必须先消除时钟域硬重置，再用新数据判断条件预算真实收益，避免多变量叠加。

## 自动验证与实机门禁

单元测试固定以下边界：240 FPS和68 FPS检测发布均使用100 ms；20 FPS使用200 ms；未知发布率回退60 FPS并使用100 ms。CSV测试要求`InferenceFPS,TrackerStaleTimeoutMs`列连续存在，原有表头/数据列数和版本身份断言继续生效。

提交前`git diff --check`通过，DML Release和CUDA Release均使用正式脚本完整清理构建成功，两套CTest各9/9通过。首次为测试直接包含`thread_loops.h`时，算法测试目标因没有鼠标与第三方跟踪器私有包含目录而编译失败；该测试只需要纯时间换算，不应扩大依赖面，因此最终把`trackerStaleTimeoutMs()`拆到无项目私有依赖的`runtime/tracker_timing.h`，主循环和测试共同复用。CUDA仅出现既有OpenCV模板及CUDA头文件本地代码页警告，无新增编译错误。

程序提交后仍须重新生成DML/CUDA干净构建，并核对UI、CSV和产物统一为同一程序提交、对应后端和r58身份。

下一轮仍只采集jump三个独立试次，每轮前清空流水线缓冲。首要门禁不是立即追求更低P95，而是：

- 25～48 ms正常观测间隔后不得再出现方向锁硬重置；
- `TrackerStaleTimeoutMs`应稳定为100 ms，`ControllerRevision=58`；
- 预测中断和冷启动段应显著低于r57的22/99/78次；
- 在预测连续后重新评估`ConditionalSpeedBudgetActive`命中、P95/P99、末1秒P95、框内率、输出步进和换向。

若三轮jump通过连续性和至少15%的P95收益门禁，再复测left/right/reverse/static各一轮；在此之前继续legacy并禁止active。

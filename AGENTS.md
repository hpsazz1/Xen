# Xen 项目开发铁律

本文件适用于仓库根目录及全部子目录。参与本项目的开发者和自动化开发代理必须遵守以下规则。

1. 每次代码变更必须提交到本地 Git 仓库并推送到云端远端，提交信息使用中文；提交前必须确认改动范围、文档和测试结果，推送后必须确认远端分支已包含该提交。
2. 所有代码修改必须同步更新相关文档；问题分析、技术选型、流程和踩坑记录统一保存在根目录 `docs/`。
3. 开发前先理解完整调用链、数据流、配置、线程和构建影响，从顶层设计推进，禁止连续堆叠局部补丁。
4. 项目定位是“小而美的瑞士军刀”：围绕核心功能设计最小闭环，避免大而全、过度设计和过度开发。
5. 每项代码、构建、配置逻辑改动必须形成“分析—设计—实现—文档—单元测试—双后端构建—版本身份核对—本地提交—云端推送”的闭环；核心逻辑必须有详细单元测试。纯非代码变更按第16条执行精简闭环。
6. 问题、规范、流程和踩坑记录必须写清背景、原因、排查过程、解决方案、验证结果和后续建议。
7. 所有方案必须基于真实使用场景、可验证数据和落地约束，禁止不切实际、假大空的设计。
8. 新增项目文档命名为 `序号中文文件名日期.md`，不得含空格，例如 `003输入延迟排查20260713.md`。历史文档暂不强制改名；`docs/todolist.md` 是待办清单的唯一固定命名例外。
9. 开发优先保障核心功能快速形成闭环；非阻塞细节记录到 `docs/todolist.md`，不得长期占用主线开发时间。
10. 技术和算法选型优先采用成熟行业最佳实践与稳定依赖；除商业机密、关键差异化能力或现有方案不适用外，不重复造轮子。选型结论必须记录依据和权衡。
11. 在适用且可用时优先使用 RTK、MCP 和项目 Skills 提高检索、分析和执行效率；工具不可用时采用可靠替代方案，不得虚构使用结果。
12. 构建和测试流程必须沉淀为正式、可复用脚本；禁止反复创建并删除一次性脚本。临时诊断命令如有复用价值，应整理进正式脚本或文档。
13. 核心代码、变量、常量和业务流程必须补充准确、详细的中文注释；注释解释意图、单位、边界和原因，不复述语法。
14. 每次正式构建必须同步更新 UI 标题、流水线 CSV 和构建产物中的版本身份；三者必须使用同一构建身份来源，并至少包含后端、Git 提交和控制器修订号。代码、构建或运行行为变更提交后必须重新构建，使最终可执行文件显示该次干净提交，不得交付 `-dirty` 或早于最近一次程序相关变更的旧版本。后续纯非代码提交不会使已经验证的构建失效，测试时核对程序 UI、CSV 与构建产物身份一致，并记录其对应的程序构建提交，不强制等于仅含非代码变更的当前 `HEAD`。
15. 代码、构建脚本、配置逻辑、项目文件或运行资源变更完成本地中文提交后，必须推送当前分支到已配置的云端远端；禁止只提交本地不推送。推送失败时必须排查并明确报告，不得把未同步状态宣称为完成；推送成功后使用远端引用核对提交一致性。
16. 纯非代码变更包括仅修改 `AGENTS.md`、`docs/*.md`、说明文字或待办清单，且不涉及源码、脚本、CMake/项目文件、配置默认值、资源、清单或运行行为。此类变更只需检查内容一致性、执行 `git diff --check` 并使用中文本地提交；无需运行单元测试、DML/CUDA构建、提交后版本重建或云端推送，也不因仓库 `HEAD` 前进而要求废弃此前基于干净程序提交构建的测试版本。用户明确要求推送时除外。

代码变更提交前最低检查：

- `git diff --check`
- 相关单元测试
- DML Release 构建
- CUDA Release 构建（涉及共享代码时）
- UI 标题、CSV 与构建产物版本身份一致，且对应本次程序相关提交
- 文档同步检查
- 中文 Git commit
- `git push` 推送当前分支到云端并核对远端提交

纯非代码变更提交前最低检查：

- 改动范围与文档内容检查
- `git diff --check`
- 中文 Git commit
